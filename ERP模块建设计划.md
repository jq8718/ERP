# ERP模块建设计划

## 1. 项目目标

建设一个面向销售、客户、库存、采购、供应商、生产指令和权限管理的 ERP 模块，用于统一管理客户资料、产品标签、包装信息、收件信息、销售订单及合同附件、库存判断、采购需求、供应商进货、退货、借样、对账、收付款和数据备份。

系统优先解决以下问题：

- 客户资料、标签备注、包装要求、地址电话等信息统一建档。
- 物料、配件、原料、成品建立统一编码档案，编码支持人工录入字母和数字。
- 销售订单审核确认后自动检查库存和欠料情况。
- 库存满足时第一期支持手动创建生产指令单，第二期支持自动生成生产指令单。
- 欠料时自动提醒，并标红显示。
- 支持客户退货、供应商退货、客户借样、借样转销售。
- 支持销售订单审核、退货审核、采购审核等审批流程。
- 支持客户收款和供应商付款记录，不直接连接银行账号。
- 不同岗位按照权限查看不同模块和数据范围。
- 每天自动备份，并支持导出 Excel 表格。

## 2. 业务范围

### 2.1 物料编码管理

支持为以下对象建立编码：

- 原材料
- 配件
- 半成品
- 成品
- 包装材料
- 客户专属产品

公司目前没有现成物料编码规则，第一期采用人工录入编码。

为减少重复和录入错误，系统可以提供“建议编码”功能：用户选择物料类别后，系统按类别前缀、日期或流水号生成一个建议值，用户仍可手动修改。建议编码只作为辅助录入，不作为公司强制编码规则。

编码规则：

```text
只允许输入英文字母和数字。
系统校验物料编码唯一性。
建议统一使用大写字母，便于查询和导入。
建议避免使用容易混淆的字符，例如 I、O、0、1。
```

示例：

```text
RMGENAL60610001A
PTC001LABEL0008A
FGC001PROD0126B
PKGENBOX0021A
```

字段建议：

| 字段 | 说明 |
| --- | --- |
| 物料编码 | 系统唯一编号 |
| 物料名称 | 原料、配件、成品等名称 |
| 物料类别 | 原材料、配件、成品、包装材料等 |
| 规格型号 | 尺寸、颜色、材质、型号等 |
| 单位 | 个、套、箱、kg、米等 |
| 库存基础单位 | 库存核算使用的基础单位 |
| 数量精度 | 该物料数量计算保留位数，例如 0、2、3 |
| 默认供应商 | 常用供应商 |
| 默认采购价格 | 库存零部件、原料、包装材料的常用采购参考价 |
| 最新采购价格 | 最近一次进货单自动回写的采购单价 |
| 采购价格税别 | 含税、未税 |
| 采购币种 | 默认人民币，可按需要扩展 |
| 安全库存 | 低于此数量时提醒 |
| 是否客户专属 | 区分通用物料和客户专属物料 |
| 状态 | 启用、停用 |
| 备注 | 其他说明 |

### 2.2 计量单位与换算

系统需要支持基础单位和业务单位换算，避免采购、库存、BOM 使用不同单位时计算错误。

单位规则：

- 每个物料需要维护一个库存基础单位，例如个、kg、米。
- 采购、BOM、销售和库存可以使用不同业务单位，但必须能换算到该物料的库存基础单位。
- 换算关系需要按物料维护，不能只按单位名称全局套用。
- 涉及库存扣减、BOM 用量计算、采购入库和成本核算时，系统统一换算为库存基础单位后计算。
- 换算结果需要按物料配置的小数精度处理。

单位换算字段建议：

| 字段 | 说明 |
| --- | --- |
| 物料编码 | 对应物料 |
| 来源单位 | 例如箱、kg、g |
| 目标单位 | 该物料的库存基础单位 |
| 换算比例 | 1 来源单位等于多少目标单位 |
| 小数精度 | 数量计算保留位数 |
| 状态 | 启用、停用 |
| 备注 | 特殊换算说明 |

### 2.3 BOM 管理

BOM 是成品和零部件、原料、包装材料之间的用量关系。成品需要进行 BOM 管理，系统中建议将 BOM 绑定到“系统成品编码”，而不是直接绑定到客户名称。

BOM 管理必须支持版本管理。不同版本 BOM 需要独立保存，历史版本不能被新版本覆盖。

成品 BOM 管理规则：

- 需要生产或销售管理的成品必须建立系统成品编码。
- 成品必须维护 BOM 后，才能在销售订单确认时进行完整的物料需求判断。
- 如果销售订单只使用已有成品库存发货，也应能追溯该成品对应的系统成品编码和 BOM 版本。

关联关系建议：

```text
客户
  -> 客户产品资料
  -> 系统成品编码
  -> 启用中的 BOM 版本
  -> BOM 明细零部件 / 原料 / 包装材料
```

这样处理的好处：

- 一个成品可以被多个客户产品引用。
- 同一个客户产品可以固定对应一个系统成品编码。
- 销售订单选择客户产品后，系统能自动找到成品和 BOM。
- 成品改版时，可以通过 BOM 版本管理，不影响历史订单。
- 客户专属标签、包装要求保存在客户产品资料中，实际要消耗的标签、包装材料可以写入 BOM 明细。

BOM 主表字段建议：

| 字段 | 说明 |
| --- | --- |
| BOM 编号 | 系统自动生成 |
| 成品编码 | BOM 对应的系统成品 |
| BOM 版本 | A、B、C 或 V1、V2 |
| 基准数量 | 生产多少个成品对应以下用量，通常为 1 |
| 生效日期 | BOM 开始使用日期 |
| 失效日期 | 可选 |
| 状态 | 草稿、待审核、已驳回、已启用、已停用、已作废 |
| 是否默认 BOM | 同一成品多个版本时，指定默认启用版本 |
| 备注 | 工艺、版本说明等 |

BOM 版本管理规则：

1. 同一个成品可以有多个 BOM 版本。
2. 同一时间只能有一个默认启用 BOM，除非按客户、工艺或订单特殊指定。
3. 已启用并被销售订单、生产指令单使用过的 BOM 不能直接覆盖修改。
4. BOM 需要修改时，应复制旧版本生成新版本，例如从 A 版复制为 B 版。
5. 新版本审核启用后，只影响后续新订单，不影响历史订单。
6. 历史销售订单和生产指令单必须记录当时使用的 BOM 编号和 BOM 版本。
7. 已停用版本不能再被新订单自动调用，但可以查询历史。
8. BOM 版本变更需要记录变更原因、变更人和变更时间。
9. 系统应支持 BOM 版本对比，显示新增、删除、用量变化的子件。

BOM 状态建议：

| 状态 | 说明 |
| --- | --- |
| 草稿 | 可编辑，未投入使用 |
| 待审核 | 已提交，等待审核 |
| 已驳回 | 审核未通过，需要修改后重新提交 |
| 已启用 | 可被销售订单和生产指令单调用 |
| 已停用 | 不再用于新订单，但保留历史 |
| 已作废 | 录入错误或不应继续使用 |

BOM 改版流程：

```text
选择旧 BOM 版本
  -> 复制为新版本
  -> 修改子件、用量、损耗率或工序
  -> 填写变更原因
  -> 提交审核
  -> 审核通过
  -> 新版本启用
  -> 旧版本停用或继续保留为非默认版本
```

BOM 操作入口建议：

- BOM 列表页和 BOM 详情页都应提供“复制为新版本”入口。
- 已启用或已停用的历史 BOM 不能直接编辑关键用量，只能复制为新版本后修改。
- 复制新版本时默认带出旧版本主表、明细、替代料、损耗率和工序，用户修改后再提交审核。
- 新版本提交审核前需要填写改版原因，便于后续追溯。

BOM 版本对比内容：

- 新增的子件。
- 删除的子件。
- 用量变化。
- 损耗率变化。
- 替代料变化。
- 工序变化。
- 包装材料或标签材料变化。

BOM 明细字段建议：

| 字段 | 说明 |
| --- | --- |
| 成品编码 | 对应 BOM 主表 |
| 子件物料编码 | 零部件、原料、包装材料编码 |
| 子件名称 | 自动带出 |
| 用量 | 生产基准数量成品所需数量 |
| 单位 | 自动带出，可按换算规则处理 |
| 损耗率 | 生产损耗，例如 2% |
| 是否必需 | 必需、可选 |
| 替代料组 | 可选，用于多个物料可替代时 |
| 使用工序 | 可选，例如组装、包装、检验 |
| 备注 | 特殊说明 |

BOM 必需 / 可选子件规则：

- 必需子件用于齐套判断，缺料时销售订单明细行库存检查状态为欠料。
- 可选子件不阻止齐套和生产，但需要在库存检查结果中显示警告。
- 可选子件库存不足时，不生成阻塞型欠料提醒；可生成普通提醒供生产或采购参考。
- 生产领料单默认带出必需子件；可选子件有库存时可以带出建议领料行，无库存时可以跳过或人工添加。
- 可选子件被人工加入领料单后，也必须遵守库存扣减和批次追溯规则。

### 2.4 BOM 输入方式

系统应支持手工录入和 Excel 导入两种 BOM 输入方式。

手工录入流程：

1. 先建立成品物料编码。
2. 再建立零部件、原料、包装材料等子件编码。
3. 进入 BOM 模块，选择成品编码。
4. 创建 BOM 版本，并填写基准数量、生效日期和状态。
5. 添加 BOM 明细行，逐项选择子件物料编码。
6. 填写每个子件的用量、单位、损耗率、替代料和工序。
7. 保存为草稿。
8. 审核通过后启用。
9. 新销售订单只调用“已启用”的 BOM；生产指令单使用销售订单已经锁定的 BOM 编号和版本。

Excel 导入字段建议：

| 字段 | 说明 |
| --- | --- |
| 成品编码 | 必填 |
| BOM 版本 | 必填 |
| 基准数量 | 必填 |
| 子件物料编码 | 必填 |
| 子件用量 | 必填 |
| 单位 | 必填 |
| 损耗率 | 可选 |
| 替代料组 | 可选 |
| 使用工序 | 可选 |
| 生效日期 | 可选 |
| 备注 | 可选 |

导入校验规则：

- 成品编码必须已经存在，并且物料类别为成品。
- 子件物料编码必须已经存在。
- 同一成品同一版本不能重复导入相同子件，除非工序不同。
- 已启用 BOM 不建议直接修改，应复制为新版本后修改。
- BOM 启用前必须至少有一条明细。
- BOM 修改需要保留历史版本，不能影响已完成或已审核订单。

示例：

| 成品编码 | BOM 版本 | 子件物料编码 | 子件用量 | 单位 | 损耗率 |
| --- | --- | --- | --- | --- | --- |
| FGC001PROD0126A | A | PTGENSCREW0001A | 4 | 个 | 0% |
| FGC001PROD0126A | A | RMGENAL60610001A | 0.25 | kg | 2% |
| FGC001PROD0126A | A | PKC001LABEL0008A | 1 | 张 | 1% |

## 3. 客户管理模块

### 3.1 客户档案

支持录入和维护客户基础资料。

字段建议：

| 字段 | 说明 |
| --- | --- |
| 客户编号 | 系统生成或手动维护 |
| 客户名称 | 公司或个人名称 |
| 客户简称 | 便于订单展示 |
| 联系人 | 主要业务联系人 |
| 联系电话 | 联系人电话 |
| 邮箱/微信 | 可选 |
| 业务员 | 归属销售人员 |
| 结算方式 | 月结、现结、预付等 |
| 结算备注 | 客户结算要求、收款说明等，不做开票模块 |
| 客户状态 | 正常、暂停、黑名单 |
| 备注 | 其他说明 |

### 3.2 客户产品资料

客户名下可建立多个产品档案，用于保存客户专属标签、包装、备注等信息。

字段建议：

| 字段 | 说明 |
| --- | --- |
| 客户产品编号 | 客户产品唯一编号 |
| 关联客户 | 所属客户 |
| 关联成品编码 | 对应系统成品 |
| 客户产品名称 | 客户侧叫法 |
| 客户产品型号 | 客户侧型号 |
| 标签备注 | 标签文字、条码、贴标位置等 |
| 包装信息 | 包装方式、每箱数量、外箱要求等 |
| 特殊要求 | 颜色、检验、运输、交付要求等 |
| 附件 | 标签图、包装图、合同附件等 |

客户产品校验规则：

- 客户产品用于销售订单时，必须关联系统成品编码。
- 未关联成品编码的客户产品可以保存为资料草稿，但不能用于提交销售订单审核。
- 关联成品编码对应的物料类别必须为成品，且状态为启用。
- 销售订单选择客户产品时，系统自动带出成品编码、标签要求、包装要求和特殊要求。
- 销售订单可以先保存草稿，即使客户产品尚未关联成品编码或成品尚未启用 BOM。
- 草稿列表和订单详情需要标注缺失项，例如“缺少成品编码”“BOM 未启用”，但提交审核时必须完成这些前置资料。

### 3.3 收件人地址备档

每个客户支持多个收件地址。

字段建议：

| 字段 | 说明 |
| --- | --- |
| 收件人 | 姓名 |
| 电话 | 手机或座机 |
| 地址 | 详细地址 |
| 默认地址 | 是否默认 |
| 地址类型 | 收货、退货、样品等 |
| 备注 | 特殊配送说明 |

### 3.4 客户退货

支持客户退货并转入成品库存。

流程：

1. 销售或仓库创建客户退货单。
2. 选择客户、原销售订单、退货产品和数量。
3. 仓库验收入库。
4. 系统更新成品库存。
5. 财务根据退货单参与对账。

退货状态：

- 待审核
- 待收货
- 已入库
- 已完成
- 已作废

客户退货审核规则：

- 客户退货单需要审核后才能进入收货和入库流程。
- 审核驳回后退回销售或仓库修改。
- 审核记录需要保存审核人、审核时间和审核意见。

### 3.5 客户对账

支持按客户、日期范围、销售订单、退货单生成对账数据。

对账周期需要同时支持：

- 按月对账。
- 按销售订单对账。
- 按客户自定义日期范围或自定义周期对账。

对账内容：

- 销售订单金额
- 已发货数量
- 客户退货金额
- 借样转销售金额
- 应收金额
- 已收金额
- 未收金额

支持导出 Excel。

客户收款管理：

- 支持登记客户收款记录。
- 收款可关联客户、销售订单、对账单或客户自定义周期。
- 支持现金、转账、支票、其他方式等收款方式记录。
- 系统只记录收款信息和凭证附件，不直接连接银行账号。
- 收款记录用于计算已收金额和未收金额。

客户收款单字段建议：

| 字段 | 说明 |
| --- | --- |
| 收款单号 | 系统自动生成 |
| 客户 | 对应付款客户 |
| 收款日期 | 实际收到款的日期 |
| 收款金额 | 本次实收金额 |
| 收款方式 | 现金、转账、支票、其他 |
| 关联销售订单 | 可关联一个或多个销售订单 |
| 关联客户对账单 | 可选 |
| 收款凭证附件 | 转账截图、收据、聊天确认等 |
| 经办人 | 财务或授权人员 |
| 状态 | 草稿、待审核、已确认、已作废、已红冲、部分红冲 |
| 备注 | 特殊说明 |

客户收款核销规则：

- 收款确认后，需要将本次收款金额分配到一个或多个销售订单或客户对账单。
- 一张销售订单可以多次收款。
- 一笔收款可以核销多张销售订单。
- 系统根据销售订单金额、客户退货金额和已核销收款金额计算未收金额。
- 收款金额未全部核销时，需要显示未分配金额。
- 已确认收款不能直接删除，只能通过作废或红冲方式更正，并保留操作记录。
- 同一销售订单被多笔收款同时核销时，系统必须使用事务和行级锁重新计算可核销余额，禁止超额核销。
- 客户收款核销明细金额合计不能超过收款单金额。
- 单个销售订单的已核销收款金额不能超过订单应收金额减客户退货金额。
- 订单应收金额指销售订单明细确认金额汇总；可核销金额 = 订单应收金额 - 已确认客户退货金额。
- 如果客户退货发生在收款核销之后，导致已核销金额超过新的可核销金额，系统需要生成待处理超收金额。
- 待处理超收金额可通过退款、转为客户预收款、或核销到同客户其他订单处理，处理方式需要财务确认。
- 超收处理必须记录原因、操作人和审批记录。

客户收款作废与红冲规则：

- 草稿、待审核或未确认收款可以作废；已确认收款原则上不直接删除。
- 已确认收款如需更正，优先使用红冲或反向调整方式。
- 红冲不是简单修改原收款单金额，而是生成一张红冲记录，关联原收款单，金额为原收款单可红冲金额的负向影响。
- 红冲后原收款单保留为已红冲或部分红冲状态，原始凭证、核销明细和审批记录不得删除。
- 红冲金额不能超过原收款单已确认且未被红冲的剩余金额。
- 红冲确认时，需要同步冲减对应核销明细；如果原收款已核销多个销售订单，需要按原核销明细生成反向核销明细。
- 红冲导致订单从已收款变为未收款或部分未收款时，订单收款状态需要重新计算。
- 红冲需要填写原因，并按审批规则决定是否需要财务主管审核。
- 作废和红冲的区别：作废用于未产生正式财务影响或允许撤销的记录；红冲用于已确认、已参与核销或已进入对账口径的记录。

客户待处理余额规则：

- `customer_credit_balances` 用于记录客户预收款、超收款、退款待处理余额等。
- 余额来源包括收款未分配金额、客户退货后产生超收、红冲或退款后的差额处理。
- 每条余额需要记录客户、来源单据、来源类型、余额金额、已使用金额、剩余金额、状态和处理原因。
- 财务可以对待处理余额执行：转预收款、核销到同客户其他销售订单、登记退款、关闭。
- 核销到其他订单时，必须使用事务锁定余额记录和目标订单，禁止余额被重复使用。
- 登记退款时，系统只记录退款信息和凭证附件，不直接连接银行账号。
- 关闭余额必须填写原因，并保留审批或确认日志。
- 余额状态建议包括：待处理、已转预收、部分使用、已用完、已退款、已关闭。

客户收款操作流程：

```text
销售订单完成或生成客户对账单
  -> 财务进入客户收款
  -> 选择客户
  -> 系统显示该客户未收款销售订单 / 对账单
  -> 输入本次收款金额和收款方式
  -> 上传收款凭证附件
  -> 将收款金额分配到一个或多个订单 / 对账单
  -> 保存并提交审核
  -> 审核通过后确认收款
  -> 系统更新已收金额、未收金额和核销记录
```

客户余额处理流程：

```text
出现收款未分配、退货超收或红冲差额
  -> 系统生成客户待处理余额
  -> 财务进入客户余额处理页面
  -> 选择转预收款 / 核销其他订单 / 登记退款 / 关闭
  -> 填写处理金额、原因和凭证附件
  -> 如需审核则提交审批
  -> 确认后更新余额使用情况和相关订单核销状态
```

### 3.6 客户借样

支持客户借样、提醒和转销售。

流程：

1. 销售创建借样单。
2. 选择客户、样品、数量、借样日期和预计归还日期。
3. 仓库出库。
4. 到期前系统提醒销售。
5. 到期后未归还则标红提醒。
6. 客户归还样品时，仓库登记归还入库。
7. 借样品种可转为销售订单。

借样归还规则：

- 借样归还需要建立归还记录，建议使用 `sample_loan_returns` 和 `sample_loan_return_items` 保存归还单和明细。
- 第一阶段归还由仓库或有权限销售人员发起，仓库确认入库；如归还样品损坏、数量差异较大，可按审批规则提交主管审核。
- 归还数量可以小于借出数量，支持部分归还；未归还数量继续保留在借样单中。
- 归还数量不能超过该借样明细的未归还、未转销售数量。
- 归还入库时需要选择库位和库存类型，系统生成库存批次或回到指定可用库存，写入库存流水 `sample_return_in`。
- 如果归还样品已损坏或不可再销售，应入到待处理或不良品库存类型，不能直接进入可销售库存。
- 归还确认、库存入库、借样归还数量、借样状态和库存流水必须在同一数据库事务内提交。
- 已逾期借样完成全部归还或转销售后，系统自动解除逾期待办；历史逾期记录仍保留。
- 部分归还后仍有未归还数量且超过预计归还日期时，借样单继续显示逾期。
- 借样归还后不能再对已归还数量转销售；如确需销售已归还样品，应重新走正常销售订单和销售出库流程。

借样归还单字段建议：

| 字段 | 说明 |
| --- | --- |
| 归还单号 | 系统自动生成 |
| 关联借样单 | 对应客户借样单 |
| 客户 | 从借样单带出 |
| 归还日期 | 实际归还日期 |
| 归还明细 | 成品编码、归还数量、库位、库存类型 |
| 样品状态 | 完好、损坏、待检、缺件 |
| 经办人 | 仓库或授权人员 |
| 状态 | 草稿、待确认、已入库、已作废 |
| 备注 | 差异原因、损坏说明等 |

借样归还操作流程：

```text
客户归还样品
  -> 仓库进入借样单或借样归还页面
  -> 选择待归还借样单和归还明细
  -> 填写归还数量、库位、样品状态和备注
  -> 系统校验归还数量不超过未归还数量
  -> 如需审核则提交审核
  -> 仓库确认归还入库
  -> 系统增加库存、写库存流水、更新借样单归还数量和状态
  -> 如果全部归还或已转销售，关闭逾期提醒
```

借样转销售规则：

- 借样转销售时需要生成正式销售订单，并走正常销售订单审核流程。
- 转销售价格第一期由销售手工输入，可参考客户产品历史销售价格。
- 借样转销售数量不能超过该借样单未归还数量。
- 借样转销售后，系统将对应样品出库转为销售出库，不再重复扣减成品库存。
- 转销售后的借样单状态更新为已转销售或部分转销售。
- 借样转销售金额进入客户对账和收款核销范围。

借样状态建议拆成“主状态 + 逾期标记”，避免“部分归还且已逾期”这类组合状态互相冲突。

借样主状态：

- 待审核
- 已出库
- 部分归还
- 已归还
- 部分转销售
- 已转销售
- 已作废

借样逾期标记：

- `is_overdue`：是否逾期。
- `overdue_days`：逾期天数。
- `overdue_status`：未逾期、即将到期、已逾期、已关闭。
- 借样主状态用于表示业务进度，逾期标记用于提醒和工作台高亮。
- 借样全部归还、全部转销售或作废后，逾期标记更新为已关闭，但历史提醒和操作记录保留。

提醒规则建议：

- 到期前 3 天提醒。
- 到期当天提醒。
- 逾期后每天提醒，并标红显示。
- 借样超期提醒通过系统消息提醒，不使用短信、邮件或企业微信作为第一期默认提醒方式。
- 第一阶段按自然日提醒，不区分周末和节假日。

系统消息展示规则：

- 登录后在顶部通知栏显示未读消息数量。
- 首页或工作台显示待办消息列表。
- 消息中心支持查看全部消息、未读消息和已处理消息。
- 借样逾期、欠料、审批待办、备份失败等提醒都进入系统消息。
- 重要消息需要支持标红或高亮显示。

首页工作台建议：

- 登录后默认进入工作台，而不是空白首页或纯菜单页。
- 工作台显示待我审批，并按超时、金额、交期或紧急程度排序。
- 工作台显示欠料预警，缺料严重或影响交期的订单需要高亮。
- 工作台显示借样逾期和即将到期提醒。
- 工作台显示今日新增销售订单、待发货订单、待入库采购单等业务概览。
- 工作台显示我的待处理任务，例如待补 BOM、待确认收款、待处理退货、待处理盘点差异。
- 不同角色看到不同工作台内容，销售、采购、仓库、财务、生产和管理员按权限展示。

系统消息使用体验规则：

- 系统消息需要区分紧急、普通和仅通知三类。
- 紧急消息包括欠料影响交期、借样逾期、备份失败、审批超时、库存扣减失败等。
- 普通消息包括到期前提醒、待确认任务、单据被驳回等。
- 仅通知消息默认不强制弹出，避免用户每天被消息打断。
- 消息需要显示来源单据、负责人、建议处理动作和一键跳转入口。
- 用户处理完来源单据后，相关消息应自动标记为已处理或提示可关闭。
- 支持稍后提醒，但高风险消息不能永久忽略。

## 4. 销售管理模块

### 4.1 销售订单与合同附件

销售订单是系统内销售业务、库存判断、生产安排、发货和对账的依据。销售合同不作为独立业务单据流转，而是作为销售订单的附件保存。

销售订单生成后，系统自动关联客户、产品、包装、标签和收件信息。

销售订单结构：

- 销售订单采用“订单头 + 订单明细行”结构。
- 一个销售订单允许包含多个成品明细行。
- 客户、收件信息、合同附件、结算信息等属于订单头。
- 客户产品、成品编码、数量、单价、金额、BOM 编号和 BOM 版本属于订单明细行。
- 库存检查、BOM 锁定、欠料计算和生产指令单生成都按销售订单明细行逐行处理。
- 如果一个订单包含多个成品行，每个成品行可以锁定不同的 BOM 编号和 BOM 版本。

销售订单头字段建议：

| 字段 | 说明 |
| --- | --- |
| 销售单号 | 系统自动生成 |
| 客户 | 关联客户 |
| 交期 | 计划交付日期 |
| 收件信息 | 从客户地址中选择 |
| 销售合同附件 | 上传纸质合同扫描件、电子合同或其他合同附件 |
| 订单总金额 | 明细行金额汇总 |
| 订单状态 | 草稿、待审核、已驳回、待 BOM 处理、已确认、生产中、已发货、已完成、已作废 |

销售订单明细字段建议：

| 字段 | 说明 |
| --- | --- |
| 明细行号 | 系统生成或按顺序生成 |
| 所属销售单 | 关联销售订单头 |
| 客户产品 | 关联客户产品资料 |
| 成品编码 | 客户产品对应的系统成品编码 |
| 数量 | 本行销售数量 |
| 单价 | 本行销售价格 |
| 金额 | 数量 × 单价 |
| 标签要求 | 从客户产品资料带出，可在订单行锁定 |
| 包装要求 | 从客户产品资料带出，可在订单行锁定 |
| 使用 BOM 编号 | 本行确认时锁定使用的 BOM |
| 使用 BOM 版本 | 本行确认时锁定使用的 BOM 版本 |
| 库存检查状态 | 未检查、库存充足、待 BOM 处理、欠料、已齐套 |
| 行状态 | 草稿、待审核、已确认、生产中、已发货、已完成、已作废 |

销售价格带出规则：

- 销售订单选择客户和客户产品后，系统优先带出该客户产品的默认销售价格。
- 如果客户产品没有默认价格，则带出该客户最近一次相同成品成交价。
- 如果没有历史成交价，则允许销售手工输入价格。
- 手工输入或修改价格时，系统需要记录修改人和修改时间。
- 低于历史价格或低于系统设置的最低销售价时，可按审批规则触发价格审批。
- 销售订单审核通过后，销售价格锁定；后续改价必须走订单变更流程。

销售订单审核规则：

- 销售订单提交后进入待审核状态。
- 审核通过后订单才可以确认，并触发库存判断、欠料判断或生产安排。
- 审核驳回后退回销售修改。
- 审核记录需要保存审核人、审核时间和审核意见。
- 销售订单确认前，所有明细行必须有客户产品和成品编码。
- 客户产品未关联成品编码时，不允许提交销售订单审核。
- 销售订单允许先保存草稿，草稿阶段可以缺少成品编码、BOM 或合同附件等非即时必填资料。
- 提交审核时系统统一校验必填项，并在页面上列出每个明细行缺少的资料。

销售录单便捷规则：

- 销售订单页面发现客户产品、收件地址或附件缺失时，应支持在当前页面快速新增或弹窗新增。
- 客户产品缺少成品编码或 BOM 未启用时，订单草稿允许保存，并提供“发起资料补全任务”入口。
- 缺失项需要在订单详情和草稿列表中明确显示，并提供一键跳转到对应维护页面。
- 销售订单复制功能需要支持从历史订单复制客户、地址、产品明细、标签包装要求和附件备注，减少重复录入。
- 复制历史订单时，BOM 和价格必须按新订单提交时重新校验，不能直接沿用已失效资料。

销售订单状态路径：

- 需要生产时：草稿 -> 待审核 -> 已确认 -> 生产中 -> 已发货 -> 已完成。
- 成品库存充足可直接发货时：草稿 -> 待审核 -> 已确认 -> 已发货 -> 已完成。
- 无启用 BOM 时：草稿 -> 待审核 -> 待 BOM 处理 -> 重新检查 -> 已确认。
- 审核驳回时：待审核 -> 已驳回 -> 草稿。

销售订单头状态汇总规则：

- 销售订单头状态由订单明细行和审批状态汇总生成，原则上不能由普通用户手工随意修改。
- 草稿、待审核、已驳回、已作废以订单头审批状态为准。
- 审核通过后，如果任一明细行库存检查状态为待 BOM 处理，则订单头状态显示为待 BOM 处理。
- 审核通过后，如果任一明细行库存检查状态为欠料，且没有明细行进入生产中或已发货，则订单头状态显示为已确认，并在订单列表显示欠料标识。
- 如果至少一条明细行处于生产中，且订单未全部发货，订单头状态显示为生产中。
- 如果所有明细行均已发货但尚未完成收款或对账确认，订单头状态显示为已发货。
- 如果所有明细行均已完成，订单头状态显示为已完成。
- 多明细行订单允许出现一行已发货、一行生产中、一行欠料的情况；订单头取“最需要处理”的业务状态，并在明细区域展示每行真实状态。
- 状态汇总优先级建议为：已作废 > 已驳回 > 待审核 > 待 BOM 处理 > 生产中 > 已发货 > 已完成 > 已确认。

销售订单变更规则：

- 销售订单审核通过后，不允许直接静默修改关键字段。
- 关键字段包括客户、客户产品、数量、单价、交期、收件信息、标签要求、包装要求、BOM 编号和 BOM 版本。
- 需要修改关键字段时，应创建销售订单变更记录，说明变更原因。
- 变更内容需要显示变更前和变更后，便于销售、生产、仓库和财务确认影响。
- 未进入生产和未发货的订单变更，可按审批规则审核后直接更新订单。
- 已生成生产指令单、已领料、已入库或已发货的订单变更，需要提示影响范围，并由有权限人员决定是否作废重建、补单、退货或调整。
- 价格、数量、交期和收件信息变更需要同步影响对账、收付款、生产计划和发货安排。
- 所有订单变更都需要保留变更人、变更时间、变更原因和审批记录。

### 4.2 库存自动判断

销售订单审核确认后，系统按明细行逐行自动判断：

- 本行成品库存是否足够。
- 本行客户产品对应的系统成品是否已经建立并启用 BOM。
- 本行生产所需配件是否足够。
- 本行生产所需原料是否足够。
- 本行包装材料是否足够。
- 本行是否需要采购。

判断结果：

| 情况 | 系统动作 |
| --- | --- |
| 成品库存充足 | 可直接安排发货 |
| 成品没有启用 BOM | 订单进入待 BOM 处理状态，提醒销售、生产或管理员先建立并启用 BOM，暂不生成生产指令单 |
| 成品不足但配件/原料齐套 | 第一期支持手动生成生产指令单，第二期可自动生成生产指令单 |
| 配件/原料不足 | 生成欠料提醒，可生成采购需求 |
| 包装材料不足 | 生成包装材料采购提醒 |

库存齐套检查算法：

```text
输入：
  sales_order_item
  production_qty = 销售订单明细需要生产的数量
  locked_bom = 销售订单明细锁定的 BOM 编号和版本

对 locked_bom 中每个 bom_item：
  1. 计算理论需求：
     theoretical_qty = bom_item.qty / bom.base_qty * production_qty

  2. 计算含损耗需求：
     required_qty_in_bom_unit = theoretical_qty * (1 + bom_item.loss_rate)

  3. 换算到库存基础单位：
     required_qty_base = convert(required_qty_in_bom_unit, bom_item.unit, material.base_unit)

  4. 按物料数量精度向上取整或按配置保留：
     required_qty_base = round_by_material_precision(required_qty_base, material.quantity_precision)

  5. 计算可用库存：
     available_qty = SUM(inventory_batches.remaining_qty)
       WHERE material_id = bom_item.material_id
         AND batch_status = in_stock
         AND remaining_qty > 0

  6. 计算缺口：
     shortage_qty = max(0, required_qty_base - available_qty)

判断：
  所有必需子件 shortage_qty = 0 => 已齐套
  任一必需子件 shortage_qty > 0 => 欠料
  可选子件 shortage_qty > 0 => 警告，不阻止生产
```

单位换算执行顺序：

- 先按 BOM 基准数量和生产数量计算理论需求。
- 再计算损耗，得到 BOM 单位下的含损耗需求。
- 再按物料单位换算规则换算到库存基础单位。
- 最后按物料数量精度处理，并与库存基础单位下的可用库存比较。
- 损耗率只作用于 BOM 用量，不作用于单位换算比例。
- 如果找不到对应单位换算规则，库存检查失败并提示维护单位换算，不能默认按 1:1 处理。

无 BOM 处理闭环：

- 销售订单审核通过但未找到启用 BOM 时，订单不进入生产中，状态调整为待 BOM 处理。
- 系统消息提醒销售、生产和有权限维护 BOM 的人员。
- BOM 建立并审核启用后，由销售或生产人员手动重新触发库存检查。
- 重新检查通过后，销售订单锁定当次使用的 BOM 编号和 BOM 版本。
- 订单锁定 BOM 后，即使后续启用了新 BOM 版本，也不影响该销售订单和由该订单生成的生产指令单。

部分缺料处理规则：

- 第一期不自动生成部分生产指令单，默认按销售订单缺口进行完整齐套判断。
- 如果生产 100 套需要 A、B、C 三种物料，A 只能满足 60 套、B 满足 100 套、C 满足 0 套，系统判定该订单未齐套。
- 欠料提醒按完成订单缺口所需数量计算，例如 A 显示缺 40 套对应数量，C 显示缺 100 套对应数量。
- 如业务需要先生产部分数量，第一期由有权限人员手工拆分销售订单或手工创建部分生产指令单。
- 第二期可扩展“允许部分生产”的参数和审批规则。

欠料显示规则：

- 欠料行标红。
- 显示缺口数量。
- 显示建议采购数量。
- 显示预计影响订单。

欠料转采购需求规则：

- 系统生成欠料提醒后，在欠料列表和销售订单详情中提供“生成采购需求”按钮。
- 用户可以选择一条或多条欠料提醒生成采购需求。
- 多条欠料可以按物料、建议供应商、需求日期合并到同一张采购需求，也可以按销售订单分别生成。
- 采购需求明细需要记录来源欠料提醒和来源销售订单明细，便于追踪是哪张订单导致缺料。
- 欠料提醒状态建议包括：未处理、已生成采购需求、部分到货、已齐套、已关闭。
- 生成采购需求后，欠料提醒状态变为已生成采购需求，不允许重复生成同一来源的采购需求，除非原采购需求被作废或关闭。
- 采购入库后系统重新计算相关销售订单明细的齐套状态；如果仍有缺口，则更新缺口数量并保留部分到货状态。
- 欠料已经解决但销售订单作废或拆分时，欠料提醒需要同步关闭并记录原因。
- 欠料状态由库存重新检查服务更新，不由采购人员手工维护数量结果。
- 任一相关物料到货但仍未全部齐套时，欠料提醒状态更新为部分到货。
- 所有必需子件齐套后，欠料提醒状态自动更新为已齐套，并向销售、生产或订单负责人发送系统消息，提示可以创建生产指令单。
- 已关闭用于业务决定不再处理该欠料，例如订单作废、订单拆分或人工确认不再生产；关闭需要权限和关闭原因。

### 4.3 生产指令单自动生成

当销售订单明细行所需配件、原料、包装材料满足生产需求时，第二期系统可自动生成生产指令单。第一期先支持手动创建生产指令单。

生产指令单必须使用销售订单已经锁定的 BOM 编号和 BOM 版本。物料需求根据销售订单锁定的 BOM 明细自动计算，不能使用后续启用的新 BOM 覆盖历史订单。

生产指令单生成规则：

- 来源为销售订单的生产指令单，必须关联销售订单明细行，并从该明细行带出成品编码、生产数量、BOM 编号、BOM 版本、标签要求和包装要求。
- 第一期开启手动生成：有权限人员在销售订单库存检查通过后点击生成生产指令单。
- 第二期再扩展自动生成：库存检查通过后系统自动创建生产指令单。
- 手动生成和自动生成使用同一套字段、同一套 BOM 锁定规则和同一套库存校验规则。
- 生产指令单中的物料需求以 BOM 明细为准，标签文字、贴标位置、包装说明等客户展示要求以客户产品资料和销售订单锁定内容为准。

生产指令单字段建议：

| 字段 | 说明 |
| --- | --- |
| 指令单号 | 系统自动生成 |
| 来源销售单 | 关联销售订单 |
| 来源销售单明细 | 关联销售订单明细行 |
| 成品编码 | 要生产的成品 |
| 生产数量 | 根据订单缺口生成 |
| 物料需求 | 根据 BOM 自动计算 |
| 使用 BOM 编号 | 从销售订单带出，锁定当次生产使用的 BOM |
| 使用 BOM 版本 | 从销售订单带出，锁定当次生产使用的 BOM 版本 |
| 标签要求 | 从客户产品资料带出 |
| 包装要求 | 从客户产品资料带出 |
| 计划开工日期 | 可维护 |
| 计划完成日期 | 可维护 |
| 状态 | 待生产、生产中、已完成、已取消 |

供应商付款核销规则：

- 付款确认后，需要将本次付款金额分配到一个或多个进货单或供应商对账单。
- 一张进货单可以多次付款。
- 一笔付款可以核销多张进货单。
- 系统根据采购入库金额、供应商退货金额和已核销付款金额计算未付款金额。
- 付款金额未全部核销时，需要显示未分配金额。
- 已确认付款不能直接删除，只能通过作废或红冲方式更正，并保留操作记录。
- 同一进货单被多笔付款同时核销时，系统必须使用事务和行级锁重新计算可核销余额，禁止超额核销。
- 供应商付款核销明细金额合计不能超过付款单金额。
- 单个进货单的已核销付款金额不能超过进货应付金额减供应商退货金额。
- 进货应付金额指进货单已确认合格入库金额；可核销金额 = 进货应付金额 - 已确认供应商退货金额。
- 如果供应商退货发生在付款核销之后，导致已核销金额超过新的可核销金额，系统需要生成待处理超付金额。
- 待处理超付金额可通过供应商退款、转为供应商预付款、或核销到同供应商其他进货单处理，处理方式需要财务确认。
- 超付处理必须记录原因、操作人和审批记录。

供应商付款作废与红冲规则：

- 草稿、待审核或未确认付款可以作废；已确认付款原则上不直接删除。
- 已确认付款如需更正，优先使用红冲或反向调整方式。
- 红冲不是简单修改原付款单金额，而是生成一张红冲记录，关联原付款单，金额为原付款单可红冲金额的负向影响。
- 红冲后原付款单保留为已红冲或部分红冲状态，原始凭证、核销明细和审批记录不得删除。
- 红冲金额不能超过原付款单已确认且未被红冲的剩余金额。
- 红冲确认时，需要同步冲减对应付款核销明细；如果原付款已核销多个进货单，需要按原核销明细生成反向核销明细。
- 红冲导致进货单从已付款变为未付款或部分未付款时，进货单付款状态需要重新计算。
- 红冲需要填写原因，并按审批规则决定是否需要财务主管审核。
- 作废和红冲的区别：作废用于未产生正式财务影响或允许撤销的记录；红冲用于已确认、已参与核销或已进入对账口径的记录。

供应商待处理余额规则：

- `supplier_credit_balances` 用于记录供应商预付款、超付款、供应商退款待处理余额等。
- 余额来源包括付款未分配金额、供应商退货后产生超付、红冲或退款后的差额处理。
- 每条余额需要记录供应商、来源单据、来源类型、余额金额、已使用金额、剩余金额、状态和处理原因。
- 财务可以对待处理余额执行：转预付款、核销到同供应商其他进货单、登记供应商退款、关闭。
- 核销到其他进货单时，必须使用事务锁定余额记录和目标进货单，禁止余额被重复使用。
- 登记供应商退款时，系统只记录退款信息和凭证附件，不直接连接银行账号。
- 关闭余额必须填写原因，并保留审批或确认日志。
- 余额状态建议包括：待处理、已转预付、部分使用、已用完、已退款、已关闭。

供应商付款操作流程：

```text
采购入库完成或生成供应商对账单
  -> 财务进入供应商付款
  -> 选择供应商
  -> 系统显示该供应商未付款进货单 / 对账单
  -> 输入本次付款金额和付款方式
  -> 上传付款凭证附件
  -> 将付款金额分配到一个或多个进货单 / 对账单
  -> 保存并提交审核
  -> 审核通过后确认付款
  -> 系统更新已付金额、未付金额和核销记录
```

供应商余额处理流程：

```text
出现付款未分配、退货超付或红冲差额
  -> 系统生成供应商待处理余额
  -> 财务进入供应商余额处理页面

  -> 如需审核则提交审批
  -> 确认后更新余额使用情况和相关进货单核销状态
```

### 5.3 采购批次价格与库存成本

由于每一批采购零部件价格可能不同，系统不能只保存一个固定采购价，需要按采购批次保存实际价格。

建议采用三层价格：

| 价格类型 | 用途 | 是否覆盖历史 |
| --- | --- | --- |
| 物料参考采购价 | 用于采购下单、报价预估、供应商比价 | 可以更新 |
| 采购单明细单价 | 记录本次采购的真实成交价 | 不覆盖历史 |
| 库存批次成本价 | 记录这批货入库后的库存成本 | 不覆盖历史 |

处理规则：

1. 每次进货录入时，必须填写本批实际采购单价。
2. 进货入库后，系统生成库存批次号，并保存本批数量、单价、税别、供应商和入库日期。
3. 同一个零部件可以存在多个库存批次，每个批次有自己的采购价格。
4. 仓库出库或生产领料时，可按批次选择，也可按先进先出自动扣减。
5. 财务成本核算可选择先进先出、移动加权平均或指定批次成本。
6. 最新采购价格可以自动回写到物料档案，但历史批次价格不能被覆盖。

批次自动推荐规则：

```text
SELECT *
FROM inventory_batches
WHERE material_id = ?
  AND location_id = ?
  AND batch_status = 'in_stock'
  AND remaining_qty > 0
ORDER BY received_at ASC, batch_no ASC
```

规则说明：

- 先进先出按入库时间 `received_at` 升序排序，同一入库时间再按批次号升序排序。
- 批次状态为冻结、作废、已用完的批次不能被自动推荐。
- 如果指定库位库存不足，可以按业务配置决定是否推荐其他库位；第一阶段默认只推荐当前选择库位。
- 自动推荐结果只是建议，仓库人员可按权限调整批次，调整需要记录原因。
- 冻结批次只能查询和盘点，不能用于销售出库、生产领料、借样出库或供应商退货出库。

冻结概念区分：

- 批次冻结是 `inventory_batches.batch_status = frozen`，属于持久状态，需要人工冻结或解冻。
- 批次冻结会影响可用库存计算、FIFO 推荐、销售出库和生产领料。
- 盘点临时冻结是盘点单范围内的临时业务锁，不直接修改批次状态。
- 盘点临时冻结只在盘点单处于盘点中时生效，盘点完成、作废或关闭后释放。
- 如果系统配置为盘点中冻结指定库位或物料，则被冻结范围内库存不参与销售出库和生产领料。
- 第一阶段库存齐套检查可以继续统计盘点临时冻结库存，但创建销售出库或生产领料时必须重新校验临时冻结范围；如果希望严格控制，可配置为齐套检查也排除盘点临时冻结库存。

库存批次与当前库存关系：

- `inventory_batches` 是批次库存和批次成本的来源表。
- `inventory` 是按物料、库位、库存类型汇总后的当前库存缓存表。
- `inventory` 的数量应等于对应 `inventory_batches` 剩余数量的汇总。
- `inventory` 建议唯一键为：物料编码 + 库位 + 库存类型。
- 入库时如果 `inventory` 已存在对应记录则更新数量，不存在则插入新记录。
- 入库、出库、移库、盘点调整时，需要在同一个数据库事务内同时更新 `inventory_batches`、`inventory` 和 `inventory_transactions`。
- 系统应定期提供库存汇总校验，发现 `inventory` 与 `inventory_batches` 汇总不一致时提醒管理员。

批次字段建议：

| 字段 | 说明 |
| --- | --- |
| 库存批次号 | 系统自动生成 |
| 物料编码 | 对应零部件、原料或包装材料 |
| 来源进货单 | 对应采购入库记录 |
| 供应商 | 本批采购供应商 |
| 入库日期 | 本批入库日期 |
| 入库数量 | 本批入库数量 |
| 剩余数量 | 本批当前剩余库存 |
| 本批采购单价 | 本批真实采购价格 |
| 价格税别 | 含税、未税 |
| 币种 | 默认人民币 |
| 批次状态 | 在库、已用完、冻结、作废 |

### 5.4 进货录入

支持采购到货后进行进货录入。

采购需求、采购单与进货单关系：

- `purchase_requests` 用于记录欠料、低库存或人工提出的采购需求。
- `purchase_orders` 用于记录向供应商正式下单的采购单。
- `purchase_receipts` 用于记录实际到货、验收和入库。
- 采购需求审核通过后，采购人员可以手动生成采购单。
- 一张采购需求可以拆分到多张采购单。
- 一张采购单可以对应多次进货单，用于分批到货。
- 采购单明细需要记录采购数量、已到货数量、未到货数量和关闭状态。
- 进货单需要关联采购单和采购单明细；无采购单的临时进货需要权限控制和原因说明。

采购单字段建议：

| 字段 | 说明 |
| --- | --- |
| 采购单号 | 系统自动生成 |
| 供应商 | 对应供应商 |
| 来源采购需求 | 可选，关联采购需求 |
| 采购日期 | 下单日期 |
| 预计到货日期 | 可选 |
| 采购总金额 | 明细行汇总 |
| 状态 | 草稿、待审核、已确认、部分到货、已到货、已关闭、已作废 |
| 备注 | 采购说明 |

采购单明细字段建议：

| 字段 | 说明 |
| --- | --- |
| 采购单 | 对应采购单 |
| 来源采购需求明细 | 可选 |
| 物料编码 | 采购物料 |
| 采购数量 | 本次下单数量 |
| 已到货数量 | 已通过进货单入库或验收的数量 |
| 未到货数量 | 采购数量 - 已到货数量 |
| 采购单价 | 本次下单价格 |
| 金额 | 采购数量 × 采购单价 |
| 行状态 | 未到货、部分到货、已到货、已关闭 |

采购单价来源规则：

- 创建采购单或进货单时，系统优先带出该物料对应该供应商的有效默认采购价格。
- 如果没有供应商价格，则带出物料档案中的最新采购价格。
- 如果最新采购价格也为空，则允许手工输入，但必须填写原因或备注。
- 最终入库批次成本以进货单明细确认的实际单价为准。
- 进货单审核确认后，系统可回写物料最新采购价格，但不能覆盖历史采购单和库存批次成本。

流程：

1. 采购需求审核通过。
2. 采购人员生成采购单并提交审核。
3. 供应商到货后，采购或仓库根据采购单创建进货单。
4. 选择供应商、采购单、物料编码、数量、单价。
5. 仓库验收。
6. 合格品入库。
7. 不良品记录并进入待处理状态。

进货单状态：

- 待审核
- 待入库
- 部分入库
- 已入库
- 已作废

采购审核规则：

- 采购相关单据需要审核后才能正式入库或参与付款对账。
- 审核范围包括采购需求、采购单、进货单或后续采购单据。
- 采购需求用于提出采购计划，审核通过后可手动生成采购单。
- 采购单用于正式下单，采购单审核通过后才能用于进货。
- 进货单用于实际到货验收和入库，进货单本身也需要审核。
- 审核记录需要保存审核人、审核时间和审核意见。

进货入库事务规则：

- 进货验收、合格品入库、库存批次生成、当前库存更新、库存流水写入必须在同一个数据库事务内完成。
- 如果任一步失败，整笔入库事务必须回滚。
- 不良品记录和待处理数量也应在同一事务中写入，避免入库数量和验收结果不一致。
- 进货单从待入库变为部分入库或已入库时，必须和库存变化在同一事务内提交。

### 5.5 不良退供应商

进货中发现不良品，可退给供应商。

流程：

1. 创建供应商退货单。
2. 选择来源进货单和不良数量。
3. 仓库出库。
4. 财务参与供应商对账。
5. 完成退货记录。

供应商退货审核规则：

- 供应商退货单需要审核后才能执行出库。
- 审核驳回后退回修改或作废。
- 审核记录需要保存审核人、审核时间和审核意见。

## 6. 库存管理模块

### 6.1 库存对象

系统采用单一仓库管理，但需要支持多库位。

库存层级：

```text
单一仓库
  -> 多个库位
  -> 物料 / 成品 / 样品 / 不良品库存
```

库存需要区分：

- 原材料库存
- 配件库存
- 包装材料库存
- 成品库存
- 样品库存
- 不良品库存

库位管理规则：

- 所有库存数量需要记录所在库位。
- 入库时需要选择或默认带出库位。
- 出库、生产领料、借样出库和退货出库时需要从指定库位扣减。
- 库位之间需要支持移库记录。
- 第一期不需要支持多仓库。
- 第一期不需要条码扫码出入库。

库存并发扣减规则：

- 销售出库、生产领料、借样出库、供应商退货出库、库位移库和盘点调整都必须使用数据库事务。
- 扣减库存前必须锁定对应物料、库位和批次库存记录，建议使用 PostgreSQL 行级锁，例如 `SELECT ... FOR UPDATE`。
- 也可在库存表中增加版本号字段，用乐观锁防止并发覆盖。
- 系统必须禁止库存扣减后出现负数，数据库层和应用层都需要校验。
- 同一笔出库需要同时写入库存批次扣减、当前库存扣减和库存流水。
- 如果多个用户同时操作同一物料库存，后提交的事务必须基于最新库存重新判断，不能使用旧的库存检查结果直接扣减。

库存预留规则：

- 第一期可以不做复杂库存预留，但销售订单确认、生产领料和销售出库之间必须重新校验库存。
- 如后续需要避免确认订单后被其他单据抢占库存，可增加库存预留字段和释放规则。

### 6.2 库存动作

系统需要记录以下库存流水：

- 采购入库
- 生产领料
- 生产入库
- 销售出库
- 客户退货入库
- 供应商退货出库
- 借样出库
- 借样归还入库
- 借样转销售
- 库位移库
- 盘点调整

### 6.3 库存预警

预警规则：

- 低于安全库存提醒。
- 销售订单导致欠料时提醒。
- 借样逾期提醒。
- 采购到货延期提醒。

### 6.4 生产领料

生产领料用于根据生产指令单从库存中领出原料、配件和包装材料。

生产领料规则：

- 生产领料必须关联生产指令单。
- 生产或仓库可以发起领料单，具体权限由角色配置控制。
- 领料明细默认由系统根据生产指令单锁定的 BOM 明细计算。
- 领料需求数量 = BOM 用量 × 生产数量 × (1 + 损耗率)。
- 系统自动计算建议领料数量，允许有权限人员按实际情况调整，调整需要填写原因。
- 领料可以一次性完成，也可以分批领料。
- 每次领料都需要从指定库位和库存批次扣减。
- 系统默认按先进先出自动推荐库位和库存批次，仓库人员只在需要时手动调整。
- 领料确认后写入库存流水，库存动作类型为生产领料。
- 已领料数量不能超过生产指令单需求数量和允许损耗范围，超额领料需要审批。

生产领料单字段建议：

| 字段 | 说明 |
| --- | --- |
| 领料单号 | 系统自动生成 |
| 关联生产指令单 | 来源生产指令单 |
| 领料部门 / 人员 | 生产或仓库人员 |
| 领料日期 | 实际领料日期 |
| 状态 | 草稿、待审核、已出库、部分出库、已作废 |
| 备注 | 特殊说明 |

生产领料明细字段建议：

| 字段 | 说明 |
| --- | --- |
| 领料单 | 对应领料单 |
| 物料编码 | 需要领用的物料 |
| 建议领料数量 | 根据 BOM 和生产数量自动计算 |
| 实际领料数量 | 本次实际领出数量 |
| 单位 | 领料单位 |
| 库位 | 从哪个库位扣减 |
| 库存批次 | 从哪个批次扣减 |
| 损耗率 | 从 BOM 带出 |
| 调整原因 | 实际领料与建议数量不一致时填写 |

生产领料流程：

```text
生产指令单创建
  -> 系统根据锁定 BOM 计算建议领料明细
  -> 生产或仓库创建领料单
  -> 系统按先进先出自动推荐库位和批次
  -> 仓库可按实际情况调整库位和批次
  -> 提交审核或确认出库
  -> 系统扣减库存批次和当前库存
  -> 写入库存流水
```

批次选择体验规则：

- 生产领料默认按先进先出自动推荐库存批次，减少人工逐项选择。
- 如果同一物料存在多个库位，优先推荐默认库位或可用数量满足本次领料的库位。
- 仓库人员只在推荐批次不足、实物不一致或需要指定批次时手动调整。
- 手动调整批次需要记录调整人和原因，便于追溯。

### 6.5 生产入库

生产入库用于生产完成后将成品转入库存，并关闭或更新生产指令单状态。

生产入库规则：

- 生产入库必须关联生产指令单。
- 生产入库可以一次性完成，也可以分批入库。
- 入库数量不能超过生产指令单生产数量和允许超产范围，超产入库需要审批。
- 入库成品必须写入成品库存批次、当前库存和库存流水，库存动作类型为生产入库。
- 生产入库时需要记录生产批次、入库库位、合格数量、不良数量和备注。
- 如果生产指令单来源于销售订单明细，入库后系统重新计算对应销售订单明细是否可发货。
- 生产入库和生产指令单状态更新必须在同一数据库事务内完成。

生产入库单字段建议：

| 字段 | 说明 |
| --- | --- |
| 生产入库单号 | 系统自动生成 |
| 关联生产指令单 | 来源生产指令单 |
| 成品编码 | 入库成品 |
| 入库库位 | 成品进入的库位 |
| 入库日期 | 实际入库日期 |
| 合格数量 | 本次合格入库数量 |
| 不良数量 | 本次生产不良数量 |
| 状态 | 草稿、待审核、已入库、已作废 |
| 备注 | 入库说明 |

生产入库流程：

```text
生产完成
  -> 生产或仓库创建生产入库单
  -> 选择生产指令单和入库库位
  -> 录入合格数量和不良数量
  -> 提交审核或确认入库
  -> 生成成品库存批次并更新当前库存
  -> 写入库存流水
  -> 更新生产指令单和销售订单明细状态
```

### 6.6 销售出库与发货

销售出库用于根据销售订单从成品库存中发货，是销售订单进入已发货状态的依据。

销售出库规则：

- 销售出库必须关联销售订单和销售订单明细行。
- 销售出库可以按订单一次性发货，也可以按明细行分批发货。
- 发货前需要重新校验成品库存，不能只依赖销售订单确认时的库存检查结果。
- 销售出库默认按先进先出推荐成品库存批次，仓库可按实际情况调整。
- 出库确认后扣减成品库存批次和当前库存，写入库存流水，库存动作类型为销售出库。
- 每个销售订单明细需要记录销售数量、已发货数量、未发货数量和发货状态。
- 销售出库完成后，系统按明细行发货情况汇总销售订单头状态。

销售出库单字段建议：

| 字段 | 说明 |
| --- | --- |
| 销售出库单号 | 系统自动生成 |
| 关联销售订单 | 来源销售订单 |
| 发货日期 | 实际发货日期 |
| 收件信息 | 从销售订单带出，可按权限修改 |
| 物流信息 | 快递、物流单号、司机或车辆信息 |
| 状态 | 草稿、待审核、已出库、已作废 |
| 备注 | 发货说明 |

销售出库明细字段建议：

| 字段 | 说明 |
| --- | --- |
| 销售出库单 | 对应销售出库单 |
| 来源销售订单明细 | 对应销售订单明细行 |
| 成品编码 | 出库成品 |
| 本次发货数量 | 本次实际发货数量 |
| 库位 | 从哪个库位扣减 |
| 库存批次 | 从哪个批次扣减 |
| 已发货数量 | 出库后累计已发货数量 |
| 未发货数量 | 销售数量 - 已发货数量 |

销售出库流程：

```text
销售订单已确认或生产入库后可发货
  -> 仓库创建销售出库单
  -> 系统带出可发货明细和收件信息
  -> 系统按先进先出推荐库位和批次
  -> 仓库确认或调整发货数量
  -> 提交审核或确认出库
  -> 扣减成品库存并写入库存流水
  -> 更新销售订单明细和订单头状态
```

### 6.7 盘点管理

盘点用于核对系统库存和实际库存，并通过审核后的盘点调整修正库存。

盘点规则：

- 仓库人员或仓库主管发起盘点单。
- 盘点范围可以按全部库存、物料类别、库位或物料筛选。
- 盘点时需要记录系统账面数量、实际盘点数量和差异数量。
- 盘点单创建时需要保存账面数量快照，后续库存变化不能自动覆盖该快照。
- 盘盈盘亏必须经过审核后才能调整库存。
- 审核通过后，系统生成盘点调整库存流水。
- 盘点调整需要同时更新库存批次、当前库存和库存流水。
- 已审核盘点单不能直接删除，只能作废或通过新的盘点调整更正。
- 第一阶段建议盘点期间不强制冻结库存，但盘点范围内发生出入库时，需要在盘点单上提示“盘点期间发生库存变化”，审核前必须重新核对差异。
- 如果业务要求严格盘点，可配置为盘点中冻结指定库位或物料，冻结范围内不能出入库，紧急出库需要主管审批。
- 盘点临时冻结不等同于库存批次冻结，不修改 `inventory_batches.batch_status`；它通过盘点单范围和状态控制。

盘点单字段建议：

| 字段 | 说明 |
| --- | --- |
| 盘点单号 | 系统自动生成 |
| 盘点范围 | 全部、库位、物料类别、指定物料 |
| 盘点日期 | 实际盘点日期 |
| 发起人 | 仓库人员或主管 |
| 状态 | 草稿、盘点中、待审核、已审核、已作废 |
| 审核人 | 审核盘点差异的人 |
| 备注 | 盘点说明 |

盘点明细字段建议：

| 字段 | 说明 |
| --- | --- |
| 盘点单 | 对应盘点单 |
| 物料编码 | 被盘点物料 |
| 库位 | 被盘点库位 |
| 库存批次 | 可选，按批次盘点时填写 |
| 账面数量 | 系统当前数量 |
| 账面快照时间 | 创建盘点单时的库存快照时间 |
| 实盘数量 | 实际盘点数量 |
| 差异数量 | 实盘数量 - 账面数量 |
| 差异原因 | 盘盈盘亏说明 |

盘点流程：

```text
创建盘点单
  -> 选择盘点范围
  -> 系统带出账面数量
  -> 仓库录入实盘数量
  -> 系统计算盘盈盘亏
  -> 提交审核
  -> 审核通过
  -> 生成盘点调整流水并更新库存
```

## 7. 权限管理

### 7.1 角色划分

建议角色：

| 角色 | 权限说明 |
| --- | --- |
| 管理员 | 查看和操作全部模块 |
| 销售 | 只查看自己的客户、销售订单、借样、退货和对账信息 |
| 销售主管 | 查看所属团队客户和订单 |
| 采购 | 查看物料、库存、采购、供应商、进货信息，不查看客户敏感信息 |
| 仓库 | 查看库存、出入库、生产指令、退货、借样出入库 |
| 财务 | 查看对账、销售金额、采购金额、退货金额 |
| 生产 | 查看生产指令单、物料需求、标签包装要求 |

### 7.2 权限规则

权限分为两类：

- 模块权限：能否进入某个模块。
- 数据权限：能否看到某些数据范围。

第一期实际权限还应细分为四类：

| 权限类型 | 用途 | 示例 |
| --- | --- | --- |
| 模块查看权限 | 能否进入模块列表、详情、导出、打印等只读入口 | `sales.view`、`purchase.view`、`inventory.view`、`production.view`、`bom.view` |
| 动作处理权限 | 能否新增、编辑、确认、作废、红冲、生成下游单据 | `sales.process`、`purchase.process`、`inventory.process`、`production.process`、`bom.process` |
| 字段权限 | 能否查看金额、采购价、客户电话、收件地址等敏感字段 | `finance.view_amount`、`masterdata.view_personal_info` |
| 数据范围权限 | 能否查看本人、团队或全部数据 | `sales.view_all` |

重点规则：

- 销售只能看到自己负责的客户。
- 只具备模块查看权限的用户可以看列表、详情、导出和打印，但不能执行新增、编辑、确认、作废等处理动作。
- 具备处理权限的用户默认可以进入对应模块，但处理动作必须再次做后端权限校验，不能只依赖菜单或按钮隐藏。
- 销售角色只配置销售权限时，登录后主导航只显示销售相关入口；直接输入采购、库存、生产、财务等 URL 也必须返回无权限。
- 采购角色可以在采购侧查看欠料提醒，并按权限从欠料生成采购需求。
- 老板或管理层只读角色可以配置多个模块查看权限，但不应自动拥有权限管理、系统运维或财务处理权限。
- 采购不能看到客户联系方式、客户价格等敏感客户信息。
- 仓库可看到发货必要信息，但不一定能看到销售价格。
- 财务可看到金额和对账信息。
- 管理员可配置人员、角色和权限。

### 7.3 登录人员级别

登录人员建议同时配置“岗位角色”和“账号安全级别”。

岗位角色决定能看哪些业务模块，账号安全级别决定账号能做多敏感的系统操作。

| 安全级别 | 人员类型 | 权限说明 |
| --- | --- | --- |
| L0 停用账号 | 离职、暂停使用人员 | 不能登录，保留历史操作记录 |
| L1 普通账号 | 销售、采购、仓库、生产等普通人员 | 只能按岗位权限使用业务模块 |
| L2 敏感账号 | 财务、销售主管、采购主管 | 可查看金额、对账、采购价格等敏感信息 |
| L3 部门管理员 | 销售主管、仓库主管、财务主管 | 可管理本部门人员和部分数据 |
| L4 系统管理员 | ERP 管理员、技术管理员 | 可配置角色、权限、备份、系统参数 |

建议规则：

- 至少保留 2 个 L4 系统管理员账号，避免唯一管理员忘记密码后无法维护系统。
- L4 管理员账号不参与日常业务录入，日常操作应使用普通业务账号。
- 财务、管理员、可导出敏感 Excel 的账号建议开启二次验证，MVP 阶段采用再次输入当前登录密码。
- 离职人员账号必须停用，不能删除历史记录。

### 7.4 密码管理

密码不能明文保存，也不能让管理员查看用户原密码。

密码保存规则：

- 数据库只保存密码哈希值，不保存明文密码。
- 推荐使用 Argon2id 进行密码哈希。
- 如果技术框架暂不支持 Argon2id，可使用 bcrypt 或 PBKDF2。
- 每个密码必须使用独立随机盐值。
- 禁止使用简单默认密码，例如 `123456`、`888888`、手机号后 6 位。
- 密码修改后，旧密码不能继续使用。
- 管理员重置密码后，用户首次登录必须强制修改密码。

密码策略建议：

| 项目 | 建议规则 |
| --- | --- |
| 最小长度 | 至少 8 位，管理员建议至少 12 位 |
| 复杂度 | 不强制固定大小写组合，但禁止常见弱密码 |
| 登录失败限制 | 连续失败 5 次后锁定 15 分钟或需要管理员解锁 |
| 密码过期 | 不建议固定周期强制改密，除非怀疑泄露 |
| 初始密码 | 系统生成一次性临时密码或一次性重置链接 |
| 管理员密码 | 强度更高，并建议二次验证 |

### 7.5 忘记密码与初始化流程

小规模 ERP 建议优先采用管理员重置流程，后续如果接入邮箱、短信或企业微信，再增加自助找回密码。

#### 7.5.1 普通员工忘记密码

流程：

1. 用户向部门主管或系统管理员申请重置密码。
2. 主管或管理员核实身份。
3. 管理员在后台点击“重置密码”。
4. 系统生成一次性临时密码或一次性重置链接。
5. 临时密码或链接只允许使用一次，并设置有效期，例如 30 分钟或 24 小时。
6. 用户使用临时密码登录后，系统强制要求修改新密码。
7. 修改成功后，系统注销该用户其他旧登录会话。
8. 系统记录操作日志：申请人、重置人、重置时间、原因。

注意：

- 管理员不能查看用户原密码。
- 不允许多人共用一个默认初始化密码。
- 不建议通过微信群长期保存临时密码。
- 临时密码发出后，应要求用户尽快修改。

#### 7.5.2 管理员忘记密码

流程：

1. 优先由另一个 L4 系统管理员重置。
2. 如果只有一个管理员，应由公司负责人和技术负责人双人确认。
3. 技术负责人通过服务器本地管理工具或数据库维护脚本生成一次性重置令牌。
4. 管理员使用重置令牌设置新密码。
5. 系统记录恢复操作日志。
6. 恢复后检查是否存在异常登录、异常导出或权限修改。

建议：

- 系统上线时必须创建至少 2 个 L4 管理员。
- 可封存一个应急管理员账号，密码分开保管，只有紧急情况启用。
- 应急账号启用后必须立即改密，并记录原因。

#### 7.5.3 新员工账号初始化

流程：

1. 管理员创建员工账号。
2. 选择岗位角色和账号安全级别。
3. 系统生成一次性初始密码或重置链接。
4. 员工首次登录必须修改密码。
5. 首次登录后确认手机号、邮箱或企业微信，用于后续找回和提醒。

#### 7.5.4 离职或岗位变更

流程：

1. 离职当天停用账号。
2. 注销该账号所有登录会话。
3. 回收导出、审批、管理员等特殊权限。
4. 岗位变更时重新分配角色和数据范围。
5. 保留历史操作记录，不删除账号历史。

历史显示规则：

- 已停用用户仍保留姓名、工号、原部门和原角色快照，便于审计。
- 历史单据中的创建人、审核人、修改人应显示原姓名，例如“张三（已停用）”，不能只显示“已删除用户”。
- 用户姓名后续如有修改，历史日志建议保存当时展示名快照，避免审计时看不懂是谁操作。
- 已停用用户不能登录、不能被分配新审批任务，但历史审批记录仍可查询。

### 7.6 账号安全审计

系统需要记录以下安全事件：

- 登录成功和失败。
- 连续失败被锁定。
- 修改密码。
- 管理员重置密码。
- 权限变更。
- 敏感数据查看。
- Excel 导出。
- 管理员登录。
- 账号停用和启用。
- Session 创建、主动退出、超时失效和管理员强制失效。

高风险操作建议提醒管理员：

- 管理员密码被重置。
- 财务或采购价格相关账号异常登录。
- 短时间内大量登录失败。
- 非工作时间大量导出 Excel。
- 离职账号尝试登录。

## 8. 数据备份与 Excel 导出

### 8.1 每日备份

系统每天自动备份一次，备份保存到本机。

建议规则：

- 每天凌晨自动备份。
- 备份文件默认保存到本机指定备份目录。
- 保留最近 30 天备份。
- 每周保留一份周备份。
- 每月保留一份月备份。
- 支持管理员手动触发备份。
- 本机备份目录需要限制访问权限，普通业务账号不能直接访问。
- 第一阶段备份保存到本机，同时建议每周至少复制一份到另一台电脑、外接硬盘或 NAS，避免本机硬盘损坏导致备份全部丢失。
- 后续如有需要，可再扩展服务器或云端异地备份。
- 每次备份完成后需要记录备份文件大小、校验值和备份结果。
- 系统需要支持管理员查看备份成功或失败记录。

备份内容：

- 客户资料
- 产品资料
- 供应商资料
- 物料资料
- 销售订单
- 销售合同附件索引和附件文件
- 采购/进货数据
- 库存流水
- 退货记录
- 借样记录
- 收款记录
- 付款记录
- 收付款凭证附件文件
- 权限配置

备份校验与恢复演练：

- 备份文件生成后需要计算校验值，例如 SHA-256。
- 每月至少进行一次备份文件可读性检查。
- 每季度建议进行一次恢复演练，可恢复到测试环境，不覆盖正式数据。
- 恢复演练需要验证数据库、附件文件、用户权限和关键业务单据是否可查询。
- 备份恢复操作必须记录操作人、时间、原因和恢复结果。

### 8.3 上线初始化与期初数据

ERP 上线前需要支持期初数据整理和导入，避免系统建好后无法顺利切换使用。

期初数据范围：

- 客户档案、客户产品资料、收件地址。
- 供应商档案、供应商采购价格。
- 物料档案、单位换算、BOM 主表和 BOM 明细。
- 期初库存数量、库位、批次、成本价。
- 未完成销售订单、未发货数量、未收款金额。
- 未完成采购单、未到货数量、未付款金额。
- 借样未归还记录。
- 已存在但仍需跟进的欠料、退货、对账和收付款记录。

初始化导入规则：

- 上线初始化导入需要使用专门模板，不能和日常导入混用。
- 期初库存导入必须记录期初批次号、库位、数量和成本价。
- 未收款、未付款余额需要能追溯到来源订单或期初余额说明。
- 导入前先在测试环境试导入，并由业务负责人核对数量和金额。
- 正式导入后需要生成初始化记录，保存导入人、导入时间、模板版本、校验结果和确认人。
- 上线后原则上不再反复修改期初数据，如发现错误，应通过调整单或经审批的初始化修正处理。

## 9. 核心业务流程

### 9.1 销售订单到生产流程

```text
客户档案建立
  -> 客户产品资料建立
  -> 销售订单生成并上传销售合同附件
  -> 销售订单审核
  -> 自动检查成品库存
  -> 自动检查配件、原料、包装材料
  -> 库存满足：第一期手动创建生产指令单或安排发货，第二期自动生成生产指令单
  -> 库存不足：欠料标红并可生成采购需求
  -> 采购入库
  -> 生产完成入库
  -> 销售发货
  -> 对账
  -> 收款登记
```

```text
创建借样单
  -> 审核
  -> 样品出库
  -> 到期提醒
  -> 归还入库 / 转销售订单 / 逾期提醒
```

```text
供应商档案建立

  -> 采购审核
```

## 10. 建议数据表

### 10.1 单号生成规则

核心业务单据需要统一单号规则，便于查询、排序和人工沟通。

建议格式：

```text
业务前缀 + 日期 + 当日流水号
```

示例：

```text
SO202606080001   销售订单
RQ202606080001   采购需求
PO202606080001   采购单
PR202606080001   进货单
RT202606080001   客户退货单
SR202606080001   供应商退货单
RC202606080001   客户收款单
PY202606080001   供应商付款单
MO202606080001   生产指令单
MR202606080001   生产领料单
PI202606080001   生产入库单
SD202606080001   销售出库单
BO202606080001   BOM 编号
```

规则：

- 单号由系统自动生成，用户不能随意修改。
- 同一业务前缀下按日期生成连续流水号。
- 第一阶段采用 `document_sequences` 表 + `SELECT ... FOR UPDATE` 生成单号。
- 生成单号需要在数据库事务内锁定对应前缀和日期的流水记录，避免并发生成重复单号。
- 单号作为业务查询字段，数据库主键仍建议使用不可变 ID。
- 作废单据的单号不能复用。
- 如果事务回滚，单号可以不消耗；如果单号已经提交，即使单据作废也不能复用。
- 不采用 PostgreSQL 独立 SEQUENCE 作为业务单号主方案，因为 SEQUENCE 回滚不回退，容易产生较多跳号。

`document_sequences` 字段建议：

| 字段 | 说明 |
| --- | --- |
| prefix | 业务前缀，例如 SO、PO、PR |
| sequence_date | 日期，例如 2026-06-08 |
| current_value | 当前最大流水号 |
| updated_at | 最后更新时间 |

单号生成步骤：

```text
begin transaction
  row = SELECT * FROM document_sequences
        WHERE prefix = ? AND sequence_date = ?
        FOR UPDATE
  如果 row 不存在则创建 current_value = 0
  current_value = current_value + 1
  保存 document_sequences
  拼接 单号 = prefix + YYYYMMDD + 四位流水
  创建业务单据
commit
```

### 10.2 通用审计字段

核心业务表需要统一保留创建、修改、审核和作废信息。

通用字段建议：

| 字段 | 说明 |
| --- | --- |
| id | 数据库主键 |
| created_by | 创建人 |
| created_at | 创建时间 |
| updated_by | 最后修改人 |
| updated_at | 最后修改时间 |
| approved_by | 审核人，可选 |
| approved_at | 审核时间，可选 |
| voided_by | 作废人，可选 |
| voided_at | 作废时间，可选 |
| version | 乐观锁版本号 |
| status | 业务状态 |

适用范围：

- 客户、供应商、物料、BOM、销售订单、采购需求、进货单、退货单、库存流水、生产指令单、收款单、付款单、审批单等核心表。
- 库存、核销、权限、敏感导出等高风险操作还需要额外写入操作日志。

### 10.3 数据一致性与并发控制

ERP 中库存、收付款核销和审批状态不能只依赖前端判断，必须在数据库事务中保证一致性。

核心规则：

- 库存扣减、采购入库、生产入库、销售出库、借样出库、退货出入库必须使用数据库事务。
- 同一事务内需要同时更新业务单据、库存批次、当前库存、库存流水和相关状态。
- 库存扣减前需要锁定对应库存记录，优先使用 PostgreSQL 行级锁。
- 收款核销和付款核销需要锁定被核销的销售订单、进货单或对账单，重新计算余额后再写入核销明细。
- 数据库层需要增加非负约束，禁止库存数量、批次剩余数量、可核销余额出现负数。
- 乐观锁版本号用于防止多人同时编辑同一业务单据。
- 作废、红冲、审批通过、审批驳回等状态变化需要校验当前状态，不能跳过流程直接修改。
- 所有关键写操作需要写入操作日志。

复合事务内部顺序：

- 所有需要锁多条库存或批次记录的服务，必须按 `material_id`、`location_id`、`batch_id` 升序加锁，降低死锁概率。
- 事务内只处理必须立即一致的数据，例如业务单据状态、库存批次、当前库存、库存流水和核销明细。
- 欠料重新检查、系统消息推送、统计汇总、导出文件生成等可延后处理的动作，原则上放在事务提交后执行。

`confirm_purchase_receipt` 建议顺序：

```text
begin transaction
  1. 锁定进货单和进货明细，校验状态和数量
  2. 按 material_id、location_id 顺序准备库存记录
  3. 新增 inventory_batches 批次记录
  4. UPSERT inventory 当前库存
  5. 写入 inventory_transactions 库存流水
  6. 更新进货单状态和入库数量
  7. 更新采购单明细已到货数量和状态
commit

after commit / background event
  8. 处理 purchase_received 事件
  9. 触发相关欠料重新检查
  10. 生成齐套或仍欠料的系统消息
```

说明：

- 欠料重新检查放在事务提交后执行，避免入库事务持锁时间过长。
- 事务提交后到欠料状态更新前，允许存在短暂延迟；页面可提示“库存已更新，欠料状态稍后刷新”。
- 如果 after commit 任务失败，需要在后台任务中重试，不影响入库事务本身。

事务后事件机制：

- 第一阶段建议增加 `pending_events` 表，用于记录事务提交后需要处理的业务事件。
- 服务方法在同一业务事务内写入事件记录，例如 `purchase_received`、`sales_shipped`、`production_received`、`stock_count_adjusted`、`payment_confirmed`。
- 如果业务事务回滚，事件记录也随事务回滚，避免处理未生效的数据。
- 事务提交后可以用 `transaction.on_commit()` 触发一次处理，但不能只依赖内存回调；后台任务仍需要定期扫描 `pending_events` 中未处理事件。
- 事件处理需要支持状态、重试次数、下次重试时间、错误原因和幂等键。
- 同一业务对象的同类事件应通过幂等键避免重复处理，例如 `purchase_received:purchase_receipt_id`。
- 欠料重检、系统消息、统计刷新、汇总校验等副作用应通过事件处理器集中维护，避免每个服务方法散落大量 after commit 逻辑。

汇总与冗余字段一致性规则：

- `inventory` 是 `inventory_batches` 的汇总缓存，必须由库存服务在同一事务内更新，并由后台任务定期校验。
- `sales_order_items.shipped_qty`、`purchase_order_items.received_qty`、`production_orders.received_qty` 等进度字段如果保存到表中，只能由对应出入库服务统一更新，页面和普通编辑接口不能直接修改。
- 对于数据量不大的列表，也可以实时从 `sales_shipment_items`、`purchase_receipt_items`、`production_receipt_items` 汇总展示，但同一字段不能同时存在两个口径。
- 第一阶段如果保存上述进度字段，需要增加定期校验任务，发现汇总字段和来源明细不一致时通知管理员。
- 应收、已收、未收、应付、已付、未付的余额口径以订单 / 进货单和核销明细计算结果为准；如后续增加财务汇总表，也必须通过服务方法更新并定期校验。

### 10.4 金额、数量与时间规则

金额、数量和时间字段需要统一规则，避免四舍五入、时区和展示差异造成对账错误。

金额规则：

- 金额和单价禁止使用浮点数保存。
- 数据库金额字段使用 `numeric` / `decimal` 类型。
- 人民币金额建议保留 2 位小数。
- 采购单价、成本单价可按业务需要保留 4 到 6 位小数。
- 金额计算结果需要明确四舍五入规则，建议采用四舍五入到分。
- 订单金额、收付款金额、核销金额需要在数据库层校验非负。

金额字段精度建议：

| 字段类型 | 数据库类型 | 说明 |
| --- | --- | --- |
| 销售单价 | `numeric(14, 4)` | 支持小数单价和后续扩展 |
| 销售金额 | `numeric(14, 2)` | 人民币金额到分 |
| 采购单价 | `numeric(14, 6)` | 支持更细采购成本 |
| 采购金额 | `numeric(14, 2)` | 人民币金额到分 |
| 库存批次成本单价 | `numeric(14, 6)` | 批次成本 |
| 收款 / 付款金额 | `numeric(14, 2)` | 实收实付 |
| 核销金额 | `numeric(14, 2)` | 应收应付核销 |

数量规则：

- 库存数量、BOM 用量、单位换算比例禁止使用浮点数保存。
- 数量字段使用 `numeric` / `decimal` 类型。
- 每个物料需要配置数量精度，例如整数、2 位小数、3 位小数。
- 库存扣减、单位换算和 BOM 计算需要按物料精度处理。
- 任何库存相关计算都不能因为前端展示四舍五入而改变后台真实数量。

数量字段精度建议：

| 字段类型 | 数据库类型 | 说明 |
| --- | --- | --- |
| 销售数量 | `numeric(14, 4)` | 展示时按物料精度 |
| 采购数量 | `numeric(14, 4)` | 展示时按物料精度 |
| 库存数量 | `numeric(14, 4)` | 后台统一保存，展示按物料精度 |
| BOM 用量 | `numeric(14, 6)` | 支持精细用量 |
| 损耗率 | `numeric(7, 6)` | 例如 0.020000 表示 2% |
| 单位换算比例 | `numeric(18, 8)` | 1 来源单位等于多少目标单位 |

精度处理规则：

- 数据库统一保留足够精度，页面展示按物料数量精度显示。
- 库存扣减和齐套判断以数据库精度计算，不以页面展示值计算。
- 物料需求计算结果需要按物料数量精度处理；第一阶段建议向上取整到该物料精度，避免少领料。

时间规则：

- 数据库时间统一保存带时区时间，建议统一保存 UTC。
- 前端展示按公司所在地时区显示，当前按 Asia/Hong_Kong 处理。
- 单据日期、收付款日期、入库日期等业务日期与系统创建时间分开保存。
- 审批超时、借样到期提醒等规则按自然日计算，第一期不区分工作日和节假日。

### 10.5 状态流转与幂等规则

核心单据需要明确状态流转，避免重复提交、跳过审批和重复扣库存。

通用状态规则：

- 前端按钮只能展示当前状态允许的动作。
- 后端必须再次校验当前状态，不能只依赖前端限制。
- 同一个动作重复提交时需要幂等处理，避免重复生成单号、重复扣库存、重复核销。
- 状态变更需要记录操作人、操作时间和操作前后状态。
- 已确认、已入库、已出库、已完成的单据不能直接物理删除。
- 更正历史数据优先使用作废、红冲或调整单方式。

错误更正入口规则：

- 已审核或已确认单据页面需要提供“更正 / 作废 / 红冲 / 调整”入口，但只对有权限人员显示。
- 录错收款或付款时，通过作废、红冲或反向调整处理，不直接删除确认记录。
- 发错库位、批次或数量时，通过库存调整、反向出入库或经审批的更正单处理。
- 领料数量错误时，通过补领、退料或调整单处理，并关联原生产领料单。
- 盘点录错时，未审核前允许修改；已审核后通过新的盘点调整更正。
- 所有更正操作都需要显示对库存、应收应付、生产和发货状态的影响提示。

建议状态流转示例：

```text
销售订单：草稿 -> 待审核 -> 已驳回 / 待 BOM 处理 / 已确认 -> 生产中 -> 已发货 -> 已完成
进货单：草稿 -> 待审核 -> 待入库 -> 部分入库 / 已入库
客户收款单：草稿 -> 待审核 -> 已确认 -> 已作废 / 已红冲
供应商付款单：草稿 -> 待审核 -> 已确认 -> 已作废 / 已红冲
生产指令单：待生产 -> 生产中 -> 已完成 / 已取消
```

### 10.6 附件与文件管理规则

销售合同、标签图、包装图、收付款凭证等附件需要统一管理。

附件规则：

- 附件文件保存在应用指定文件目录，数据库只保存文件元数据和访问路径。
- 附件上传需要记录上传人、上传时间、来源单据、文件名、文件大小、文件类型和校验值。
- 附件文件名需要由系统重命名，避免用户原文件名造成路径冲突或安全风险。
- 支持的文件类型第一期建议包括 PDF、JPG、PNG、XLSX、DOCX。
- 单个附件大小需要限制，第一期建议不超过 20 MB。
- 上传文件需要校验扩展名和实际文件类型，避免伪装文件。
- 如部署环境允许，附件上传后建议进行病毒或恶意文件扫描。
- 附件访问必须走系统权限校验，不能直接暴露本机文件目录。
- 删除附件时建议逻辑删除，保留历史记录；已确认单据的附件删除需要权限或审批。
- 备份必须包含附件文件和附件元数据。

附件下载架构：

- 用户下载附件时应访问系统下载地址，例如 `/attachments/{id}/download/`，不能直接访问服务器本机文件路径。
- Django 下载视图负责校验登录、模块权限、字段权限、来源单据权限和附件状态。
- 内网小规模部署可以由 Django 视图直接返回文件流，但需要限制并发和文件大小。
- 如果使用 Nginx 或 Apache 等反向代理，建议采用 `X-Accel-Redirect` 或 `X-Sendfile`：Django 校验权限后返回内部转发头，由 Web 服务器读取文件并返回，减少 Django worker 被大文件下载长时间占用。
- 反向代理中的附件目录必须配置为 internal / protected location，外部用户不能绕过 Django 直接访问。
- 附件下载需要写入访问日志，敏感附件例如收付款凭证、合同扫描件、客户联系方式附件需要记录查看人和时间。

### 10.7 Excel 导入导出规则

Excel 是报表和批量录入工具，不能替代数据库。

Excel 导入规则：

- 导入前先生成导入任务记录，保存导入人、导入时间、文件名和导入模块。
- 导入时先做数据校验，不通过时生成错误清单，不直接写入正式数据。
- 导入成功的数据需要记录来源导入任务。
- 导入需要支持预览确认，确认后再写入数据库。
- 导入写入必须使用事务，部分失败时按模块规则决定整体回滚或只导入有效行。
- BOM、物料、客户资料等导入模板需要固定列名和版本号。

Excel 导出规则：

- 敏感数据导出需要权限控制和二次验证。
- 导出记录需要保存导出人、导出时间、导出模块、筛选条件和文件名。
- 导出客户、价格、对账、收付款等敏感数据时，建议在表格中增加导出人和导出时间水印列。
- 导出 Excel 时需要处理以 `=`、`+`、`-`、`@` 开头的文本，避免公式注入。
- 大数据量导出需要分页或后台任务，避免页面超时。
- 导出的 Excel 只作为报表，不作为主数据库。

### 10.8 列表、搜索与批量操作规则

ERP 用户大量时间在列表页查询和处理单据，列表页必须按实际工作习惯设计。

通用列表规则：

- 核心列表需要支持按单号、客户、供应商、物料编码、产品名称、状态、日期范围、负责人进行搜索或筛选。
- 销售订单、采购单、进货单、生产指令单、出入库单、盘点单、收付款单需要支持状态标签和快速筛选。
- 支持保存个人常用筛选条件，例如“我的待审核”“本周待发货”“欠料订单”“待入库采购单”。
- 列表列宽、显示列、排序字段可按用户保存偏好。
- 列表需要显示下一步可执行动作，例如提交审核、生成采购需求、创建领料单、创建出库单、确认收款。
- 批量操作需要权限控制和二次确认，例如批量导出、批量生成采购需求、批量标记消息已处理。
- 批量操作完成后需要显示成功、失败和失败原因清单。
- 无数据时需要提示可能原因，例如无权限、筛选条件过窄、暂无数据，而不是只显示空白页。

搜索体验规则：

- 客户、供应商、物料、成品和订单选择框需要支持模糊搜索。
- 搜索结果应同时显示编码、名称、规格、客户简称或供应商简称，避免选错。
- 已停用资料默认不出现在新单据选择框中，但历史单据仍可查看。

### 10.9 打印与纸面流转规则

第一期不使用条码扫码时，仓库、生产和发货更依赖纸面单据。

打印单据建议：

- 销售出库单 / 发货单。
- 送货单。
- 生产领料单。
- 生产入库单。
- 盘点表。
- 采购单。
- 进货验收单。
- 客户退货单和供应商退货单。

打印规则：

- 打印模板需要显示单号、日期、客户或供应商、物料编码、名称、规格、数量、单位、库位、批次和备注。
- 仓库用拣货类单据应按库位排序，减少来回找货。
- 对外送货单不显示内部采购成本、敏感利润和无关审批信息。
- 打印单据需要显示打印时间和打印人。
- 已作废单据再次打印时必须显示“已作废”标识。
- 打印模板应支持 A4 普通打印机，第一期不要求专用标签打印机。

### 10.10 使用权限与无权限提示规则

权限限制不能让用户误以为数据丢失。

使用提示规则：

- 用户无权限访问模块时，菜单可以隐藏或显示无权限提示。
- 用户能进入模块但看不到某条数据时，需要提示“无权限查看或数据不存在”，并避免泄露敏感信息。
- 用户尝试执行无权限动作时，需要提示缺少的权限类型，例如“缺少销售订单审核权限”。
- 如果某操作需要上级审批或管理员授权，页面应提供发起申请或联系管理员的提示。
- 被权限隐藏的金额、联系方式、附件需要显示脱敏占位，不应造成表格错位或空白误解。

### 10.11 移动端与设备使用边界

第一期以桌面浏览器为主，兼顾手机审批。

设备规则：

- 销售、采购、财务、管理员日常录入以电脑浏览器为主。
- 审批页面需要适配手机浏览器，支持查看单据摘要、附件和审批意见。
- 仓库、生产、盘点第一期可先按电脑或平板浏览器使用，不强制开发独立 App。
- 如果现场需要平板盘点或发货确认，页面按钮和表格需要适配触控操作。
- 第一阶段不做离线操作，断网时应提示保存失败或稍后重试，不能让用户误以为操作成功。

### 10.12 开发环境、配置与发布规则

开发过程需要区分环境和配置，避免测试数据、密码或密钥进入正式环境。

环境建议：

| 环境 | 用途 |
| --- | --- |
| 开发环境 | 开发人员本地开发和调试 |
| 测试环境 | 功能测试、恢复演练和用户试用 |
| 正式环境 | 日常业务使用 |

配置规则：

- 数据库密码、加密密钥、备份目录、邮件或消息配置不能写死在代码中。
- 不同环境使用不同数据库、不同密钥和不同配置文件。
- 正式环境需要关闭调试模式。
- 系统启动时需要检查关键配置是否完整。
- 依赖包版本需要锁定，避免不同电脑安装出不同版本。

发布规则：

- 数据库结构变更必须通过迁移脚本执行，不能手工改表后不留记录。
- 每次发布前需要备份数据库和附件。
- 发布前需要在测试环境执行迁移和核心流程测试。
- 发布后需要检查登录、销售订单、库存、进货、收付款、备份任务是否正常。
- 重大版本发布需要记录版本号、发布时间、发布人和变更摘要。

### 10.13 测试规则

ERP 测试需要覆盖业务流程、权限、事务和异常场景。

测试类型：

- 单元测试：编码规则、金额计算、单位换算、BOM 用量计算。
- 集成测试：销售订单确认、库存检查、进货入库、库存扣减、收付款核销。
- 权限测试：销售、采购、仓库、财务、管理员的数据可见范围。
- 并发测试：同一库存并发扣减、同一订单并发核销、同一单号并发生成。
- 恢复测试：数据库和附件备份能恢复到测试环境。
- 导入导出测试：Excel 导入校验、导出权限和敏感数据记录。

上线前最少需要通过以下端到端流程：

```text
客户建档 -> 物料建档 -> BOM 建档 -> 销售订单审核 -> 库存检查 -> 手动生产指令单
供应商建档 -> 进货审核 -> 验收入库 -> 批次库存 -> 生产领料 / 销售出库
客户退货 -> 入库 -> 客户收款 -> 按订单核销
采购入库 -> 供应商付款 -> 按进货单核销
借样出库 -> 到期系统消息 -> 归还 / 转销售
备份 -> 测试环境恢复
```

### 10.14 技术架构建议

第一期建议采用小团队易维护的单体 Web 应用，优先保证业务闭环、数据一致性和部署简单。

推荐技术栈：

| 层级 | 建议方案 | 说明 |
| --- | --- | --- |
| 后端 | Python Django | 自带用户、权限、后台管理、ORM 和表单生态，适合快速开发内部 ERP |
| 前端 | Django 模板 + 少量 Vue 或 HTMX | 第一期开发表单和列表效率高，后续复杂交互再局部增强 |
| 数据库 | PostgreSQL | 正式 ERP 推荐 |
| ORM | Django ORM | 和后端框架统一，减少手写 SQL，但库存锁等关键场景可使用事务和原生 SQL |
| 文件存储 | 本机文件目录 | 保存合同附件、收付款凭证和备份文件，目录需做权限控制 |
| 部署方式 | 单体应用 + PostgreSQL | 可部署在本机、内部服务器或云服务器 |
| 进程服务 | Windows 服务或 Linux systemd | 根据实际服务器系统选择 |

选择理由：

- 第一阶段业务表单、审批、权限和导出较多，Django 开发效率高。
- 单体应用减少前后端分离带来的部署和接口维护成本。
- ORM 可覆盖大多数增删改查，库存扣减、核销等关键事务可显式使用数据库事务和行级锁。
- 后续如果移动端或复杂前端需求增加，可逐步增加独立前端。

生产部署拓扑建议：

第一期正式环境建议按以下组件规划，避免开发完成后上线时临时决定目录、服务和访问方式。

| 组件 | 建议方案 | 说明 |
| --- | --- | --- |
| ERP Web 应用 | Django 单体服务 | 承载页面、权限、业务动作和局部 JSON 接口 |
| 数据库 | PostgreSQL 服务 | 和 Web 应用可在同一台服务器，也可独立部署 |
| 反向代理 / HTTPS 入口 | 内网可选，公网或远程审批必需 | 可使用 Nginx、Caddy 或其他成熟反向代理 |
| 静态文件目录 | `collectstatic` 输出目录 | CSS、JS、图片等静态资源由 Web 服务器或反向代理提供 |
| 附件目录 | `MEDIA_ROOT` / `attachment_dir` | 合同、凭证、标签图、包装图等业务附件 |
| 备份目录 | `backup_dir` | 数据库转储、附件备份包和校验文件 |
| 日志目录 | 应用日志、错误日志、任务日志 | 用于排查登录、任务失败、权限异常和系统错误 |
| 定时任务 | Django 管理命令 + 计划任务 | Windows 任务计划程序或 Linux cron / systemd timer |

部署规则：

- 正式环境不建议部署在经常关机或多人日常办公使用的个人电脑上，优先使用固定服务器、固定办公主机或云服务器。
- Web 应用、PostgreSQL、附件目录、备份目录和日志目录需要明确物理位置和访问权限。
- 公网或远程访问只开放 ERP Web 服务，不开放 PostgreSQL 数据库端口、附件原始目录和备份目录。
- 附件目录和备份目录不能放在 Web 静态文件目录下，所有附件访问必须经过系统权限校验。
- 第一阶段不强制 Docker 部署；如开发团队熟悉 Docker，可在开发或测试环境使用 Docker Compose，但正式备份路径、附件路径和数据库持久化目录必须清楚配置。

生产运行方式建议：

| 服务器系统 | 推荐方式 | 说明 |
| --- | --- | --- |
| Windows | Waitress + NSSM / Windows 服务 + 任务计划程序 | 适合公司内固定 Windows 主机，服务需要开机自启动 |
| Linux | Gunicorn + systemd + Nginx / Caddy + cron 或 systemd timer | 适合云服务器或内部 Linux 服务器，远程访问和 HTTPS 更方便 |

运行要求：

- Web 服务需要支持开机自启动，并在异常退出后自动重启。
- 发布时需要执行数据库迁移、静态文件收集和核心流程检查。
- 定时任务和 Web 服务应使用同一套正式配置，但不能在代码中写死数据库密码、密钥、附件目录和备份目录。
- 如果使用反向代理，Django 需要正确配置真实客户端 IP、HTTPS 转发头和允许访问的域名。

Windows 10 内网无域名部署补充规则：

- 员工使用浏览器访问 ERP，默认地址为 `http://服务器固定IP:8000/`。
- 服务器 IP 可以由安装程序自动检测，但正式使用前应设置固定 IP 或 DHCP 地址保留；否则重启后 IP 改变会导致客户端收藏地址和 `.env` 中的允许访问地址失效。
- Windows 内网轻量安装包应包含 ERP 源码、`ERP-Setup.exe`、`ERP-Uninstall.exe`、控制台启动器、安装脚本、`.env` 模板、一键安装手册、离线 Python 依赖 wheels 和 `nssm.exe`。
- Python、PostgreSQL、Git 等大安装程序可以保存在开发机 `installer/packages` 目录，不随正式 `ERP安装包` 分发；现场干净电脑可按手册单独安装。
- 安装器需要创建 `installer/logs`，`ERP-Setup.exe` 和 PowerShell 安装脚本都要写日志；安装闪退时必须能根据日志排查。
- 安装器应优先从 `installer/wheels` 离线安装 Python 依赖，现场服务器没有互联网时也能完成安装。
- Windows 服务推荐使用 Waitress + NSSM 注册为开机自启动服务，标准监听地址为 `0.0.0.0:8000`。
- 安装服务后应自动创建 Windows 防火墙入站规则，放行 TCP `8000`；如 PowerShell 防火墙命令不可用，应回退到 `netsh`。
- 安装包不应包含 `.env`、开发数据库、测试目录、历史日志、备份目录、上传附件目录、`ERP模块建设计划.md` 和安装包自身压缩文件。
- `ERP-Setup.exe` 只是启动器，不是完整程序本体；复制到服务器时必须复制整个 `ERP安装包` 文件夹，不能只复制 exe。

登录会话建议：

- 第一阶段采用服务端 Session，不优先使用纯 JWT。
- Django Session 存储后端采用数据库 Session，即 `django.contrib.sessions`。
- Session 过期时间建议为 8 到 12 小时，可配置。
- 用户主动退出、账号停用、密码修改、管理员重置密码后，需要立即注销该用户已有 Session。
- 高敏操作需要再次验证当前登录密码。
- 系统记录登录成功、登录失败、退出、Session 失效和异常登录事件。

Django Session 实现要求：

- 启用 `django.contrib.sessions` 并使用数据库表保存 Session。
- 用户停用、密码修改或管理员强制下线时，需要删除或失效该用户相关 Session。
- 如需记录设备和 IP，可在 `user_sessions` 表中保存 Session Key、用户、IP、User-Agent、创建时间、最后访问时间和失效时间。
- 不采用纯 Cookie Session 保存业务登录状态，避免无法服务端强制注销。

二次验证实现：

- MVP 阶段采用“再次输入当前登录密码”作为二次验证方式。
- 适用场景包括管理员操作、权限修改、敏感 Excel 导出、财务收付款确认、采购价格修改、BOM 启用等。
- 短信、邮件验证码和 TOTP 不作为第一期默认实现。
- 第二期可扩展 TOTP，例如 Google Authenticator 或 Microsoft Authenticator。

容量与性能预估：

| 项目 | MVP 预估 |
| --- | --- |
| 并发用户 | 5 到 30 人 |
| 注册账号 | 10 到 100 个 |
| 物料数量 | 1,000 到 50,000 条 |
| 客户数量 | 100 到 5,000 个 |
| 销售订单 | 每日 10 到 500 单 |
| 库存流水 | 每日 100 到 5,000 条 |
| 附件规模 | 第一阶段按本机磁盘容量规划 |

性能要求：

- 常用列表查询需要分页。
- 客户、物料、销售订单、库存、进货单等高频查询字段需要建立索引。
- 库存扣减、核销、审批等关键操作必须在事务内完成。
- 备份策略需要同时考虑数据库大小和附件目录大小。
- Django 列表页需要避免 N+1 查询，常用外键展示应使用 `select_related` 或 `prefetch_related`。
- 首页工作台、库存汇总、欠料统计和对账统计等聚合查询应封装在查询层，数据量增大后可改为汇总表或后台定时刷新。
- 大数据量报表和导出不在页面请求中同步完成，应进入后台任务。

运行安全基线：

- 正式环境必须关闭 `DEBUG`。
- `SECRET_KEY`、数据库密码、加密密钥、备份密钥不得提交到代码仓库。
- `ALLOWED_HOSTS`、`CSRF_TRUSTED_ORIGINS` 需要按正式域名或内网地址配置。
- 所有写操作必须启用 CSRF 防护。
- 使用 HTTPS 时，Session Cookie 需要启用 `HttpOnly`、`Secure` 和合适的 `SameSite` 策略。
- 登录失败需要限流或锁定策略，避免密码被暴力尝试。
- Django 管理后台只允许管理员访问，正式环境建议限制访问来源或更换默认入口。
- 文件上传需要限制扩展名、MIME 类型和文件大小，禁止上传可执行脚本类文件。

运行监控与健康检查：

- 系统需要记录应用错误日志、登录异常日志、后台任务日志、备份日志和关键业务审计日志。
- 慢请求和慢查询需要记录，第一阶段可先记录超过 2 秒的页面请求或超过 1 秒的数据库查询。
- 后台任务失败、备份失败、附件目录不可写、磁盘空间不足需要生成管理员系统消息。
- 建议提供管理员健康检查页面，显示数据库连接、附件目录可写、备份目录可写、最近一次备份、最近一次后台任务、磁盘剩余空间和最近错误。
- 日志文件需要定期轮转，避免长期运行后占满磁盘。

扩展路径：

- 第一阶段坚持单体架构，先保证业务闭环和数据一致性。
- 后台任务量增加、任务重试要求变复杂后，再引入 Celery、Redis 或 Django-Q。
- 附件规模变大后，可从本机目录迁移到 NAS 或对象存储，但数据库中仍只保存附件元数据和访问引用。
- 移动端需求增加后，可在现有 Django 权限和服务方法基础上增加 API，不建议第一期直接做前后端完全分离。
- 报表复杂或数据量明显增长后，可增加汇总表、只读报表账号或独立报表库。

HTTPS 与证书：

- 只在公司内网使用时，可先使用内网 HTTP 或自签名 HTTPS，但正式环境仍建议 HTTPS。
- 需要远程审批或公网访问时，必须配置域名和有效 HTTPS 证书。
- 有固定域名时，优先使用 Let's Encrypt 自动签发和续期。
- 没有固定公网 IP 时，可考虑云服务器、VPN、零信任访问或动态域名方案。
- 自签名证书不建议用于公网远程审批。

### 10.15 数据库选择

正式系统建议使用 PostgreSQL。

选择 PostgreSQL 的原因：

- 适合 ERP 这类多表、多流程、多权限、多库存流水的业务系统。
- 事务能力强，适合采购入库、销售出库、库存扣减、退货、对账等需要保证数据一致性的操作。
- 支持复杂查询，方便后续做库存报表、客户对账、采购价格历史、批次成本分析。
- 支持 JSON 字段，可用于保存标签备注、包装要求、扩展配置等灵活信息。
- 支持索引、外键、唯一约束和数据校验，能减少库存和订单数据出错。
- 备份和恢复能力成熟，适合每日自动备份。
- 后续可以支持多用户同时操作。

数据库使用建议：

| 场景 | 建议数据库 | 说明 |
| --- | --- | --- |
| 原型演示 / 单机测试 | SQLite | 开发快，但不建议正式 ERP 长期使用 |
| 正式 ERP 系统 | PostgreSQL | 推荐方案 |
| 公司已有 MySQL 服务器 | MySQL | 也可以使用，但需要提前统一技术栈 |

推荐结论：

```text
开发原型：SQLite 或 PostgreSQL
正式上线：PostgreSQL
```

正式上线时建议同时配置：

- 自动每日数据库备份。
- 数据库账号分权限管理。
- 关键表建立索引。
- 库存、订单、采购、退货相关操作使用事务。
- 定期导出 Excel 只作为报表，不作为主数据库。

### 10.16 PostgreSQL 安装建议

小规模 ERP 可以先安装在本机或一台内部服务器上。Windows 环境建议使用 PostgreSQL 官网 Windows 下载页提供的 EDB 图形安装包。

安装准备：

- 选择 64 位 Windows 安装包。
- 优先选择当前稳定版本。
- 准备一个数据库超级管理员密码，并妥善保存。
- 确认默认端口 `5432` 没有被其他程序占用。
- 正式使用时建议安装到服务器或固定办公电脑，不建议安装在经常关机的个人电脑上。

安装步骤：

1. 打开 PostgreSQL 官方下载页：`https://www.postgresql.org/download/windows/`
2. 点击 Windows installer 页面中的 `Download the installer`。
3. 下载并运行 64 位安装程序。
4. 组件建议选择：
   - PostgreSQL Server
   - pgAdmin
   - Command Line Tools
5. 设置安装目录，例如：`C:\Program Files\PostgreSQL\版本号`
6. 设置数据目录，例如：`D:\PostgreSQL\data`，正式使用时建议放在容量充足的磁盘。
7. 设置数据库管理员 `postgres` 的密码。
8. 端口保持默认 `5432`。
9. Locale 建议选择 `C` 或 `en_US.UTF-8`，避免中文系统默认排序和编码带来跨环境差异。
10. 完成安装后打开 pgAdmin，使用 `postgres` 用户和安装时设置的密码连接。

ERP 数据库初始化建议：

```sql
CREATE DATABASE erp_db;
CREATE USER erp_app WITH PASSWORD '请替换为强密码';
\c erp_db
GRANT CONNECT ON DATABASE erp_db TO erp_app;
GRANT USAGE, CREATE ON SCHEMA public TO erp_app;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO erp_app;
GRANT USAGE, SELECT, UPDATE ON ALL SEQUENCES IN SCHEMA public TO erp_app;
```

账号建议：

- `postgres`：只给管理员维护数据库使用。
- `erp_app`：ERP 系统连接数据库使用。
- `erp_readonly`：只读报表账号，可选。
- 不建议 ERP 程序长期使用 `postgres` 超级管理员账号连接数据库。
- 正式上线时应按实际迁移工具和 ORM 权限需要进一步收紧数据库权限。

安装后需要检查：

- PostgreSQL 服务是否已启动。
- pgAdmin 是否能正常连接。
- `erp_db` 数据库是否创建成功。
- ERP 程序能否使用 `erp_app` 账号连接。
- 是否已配置每日自动备份。

### 10.17 数据分级与加密管理

数据库安全建议采用“数据分级 + 权限控制 + 加密存储 + 操作审计”的方式管理。

#### 10.17.1 数据分级

| 级别 | 数据类型 | 示例 | 管理要求 |
| --- | --- | --- | --- |
| L1 普通数据 | 低敏业务数据 | 物料名称、单位、产品分类、普通备注 | 登录后可按模块权限查看 |
| L2 内部数据 | 公司内部经营数据 | 库存数量、BOM、供应商名称、采购记录、生产指令 | 按岗位授权查看 |
| L3 敏感数据 | 客户、价格、财务、联系方式 | 客户电话、收件地址、销售价格、采购价格、对账金额、收付款凭证附件 | 字段级权限、脱敏显示、必要字段加密 |
| L4 高敏数据 | 账号、密钥、备份、系统凭证 | 用户密码、数据库密码、备份密钥、接口密钥 | 不明文保存，必须加密或哈希，严格限制管理员访问 |

#### 10.17.2 加密策略

| 位置 | 策略 | 说明 |
| --- | --- | --- |
| 数据传输 | TLS/HTTPS 加密 | 浏览器、后端、数据库连接都应使用加密连接 |
| 数据库存储 | 磁盘或数据库实例加密 | 防止服务器硬盘或数据库文件被直接复制后泄露 |
| 敏感字段 | 字段级加密 | 客户电话、地址、收付款凭证附件、接口密钥等可加密保存 |
| 用户密码 | 哈希保存 | 密码不能加密后可还原，应使用安全哈希算法保存 |
| 备份文件 | 独立加密 | 每日数据库备份和导出的备份包必须加密 |
| Excel 导出 | 权限控制和水印 | 敏感数据导出需要授权，导出文件记录操作人和时间 |

#### 10.17.3 字段级权限与脱敏

敏感字段不一定所有人都能看到完整内容。

示例规则：

| 字段 | 销售 | 采购 | 仓库 | 财务 | 管理员 |
| --- | --- | --- | --- | --- | --- |
| 客户电话 | 仅自己的客户可见 | 不可见 | 可脱敏查看 | 可见 | 可见 |
| 收件地址 | 仅自己的客户可见 | 不可见 | 发货时可见 | 可见 | 可见 |
| 销售价格 | 仅自己的订单可见 | 不可见 | 不可见 | 可见 | 可见 |
| 采购价格 | 不可见 | 可见 | 可脱敏或不可见 | 可见 | 可见 |
| 对账金额 | 仅授权可见 | 不可见 | 不可见 | 可见 | 可见 |
| 收付款凭证附件 | 仅授权可见 | 授权可见 | 不可见 | 可见 | 可见 |

脱敏显示示例：

```text
手机号：138****5678
地址：江苏省苏州市***路***号
付款备注：转账凭证 **** 8888
```

#### 10.17.4 密钥管理

加密密钥不能直接写在代码里，也不能和数据库备份放在同一个目录。

建议规则：

- 主密钥由管理员单独保管，或放在服务器密钥管理服务中。
- 数据库中只保存加密后的数据，不保存明文密钥。
- 不同环境使用不同密钥，例如开发、测试、正式环境分开。
- 支持定期轮换密钥。
- 离职人员账号禁用后，应立即回收密钥和系统权限。
- 备份恢复需要密钥审批，避免任何人拿到备份文件就能恢复敏感数据。

密钥保管与恢复流程：

- 第一阶段可采用离线密钥保管方式，例如将恢复密钥打印后密封保存，或保存到加密 U 盘。
- 密钥至少由两名负责人分开保管，例如公司负责人和技术负责人。
- 任何单个人不应同时持有备份文件和恢复密钥的完整访问权。
- 管理员离职时，需要由公司负责人和技术负责人确认密钥交接，并记录交接时间和原因。
- 需要恢复备份时，先在 ERP 中提交恢复审批或形成书面审批记录。
- 审批通过后，由技术负责人取出密钥，在测试环境或正式恢复环境执行恢复。
- 恢复完成后，应重新封存密钥，并记录恢复人、审批人、恢复时间、恢复范围和恢复结果。
- 如果怀疑密钥泄露，需要立即停用旧密钥，重新加密敏感字段和备份文件。

#### 10.17.5 PostgreSQL 实现建议

如果正式数据库使用 PostgreSQL，建议采用以下方式：

- 数据库连接启用 SSL/TLS。
- 服务器磁盘或数据库实例启用静态加密。
- 普通业务字段用数据库权限和应用权限控制。
- 高敏字段可以使用应用层加密后再写入数据库，或使用 PostgreSQL 加密扩展。
- 精确查询字段可以额外保存不可逆哈希索引，例如手机号查询使用手机号哈希，不直接用明文手机号查询。
- 加密字段不适合做模糊搜索和排序，需要提前设计查询方式。

- 谁修改了 BOM。
- 谁修改了库存。

- 非财务人员查看对账金额。
- 非采购人员查看采购价格。

## 12. 审批流与远程审批

### 12.1 审批范围

以下业务需要支持审批：

- 销售订单确认。
- 客户退货。
- 供应商退货。
- 采购需求和采购相关单据。
- 采购价格修改。
- BOM 新建、改版、启用、停用。
- 库存调整。
- 借样出库和借样转销售。
- 借样归还差异处理，例如损坏、缺件或归还数量异常。
- 客户收款确认、收款红冲和客户待处理余额关闭。
- 供应商付款确认、付款红冲和供应商待处理余额关闭。
- Excel 敏感数据导出。
- 权限变更。

### 12.2 审批规则配置

审批规则用于决定某类单据由谁审批、按什么顺序审批、是否按金额或敏感程度分级。

审批规则配置方式：

- 支持按单据类型配置审批规则，例如销售订单、采购单、进货单、客户退货、供应商退货、BOM、库存调整、收付款确认等。
- 支持按角色指定审批人，例如销售主管、采购主管、仓库主管、财务、管理员。
- 支持按具体人员指定审批人。
- 支持按条件选择审批路径，例如金额、部门、业务员、供应商、客户、物料类别、是否敏感导出。
- 支持多级顺序审批，例如先部门主管，后财务，最后管理员。
- 支持审批转交和加签，但高敏事项不建议自动代理。

审批规则字段建议：

| 字段 | 说明 |
| --- | --- |
| 规则编号 | 系统自动生成 |
| 单据类型 | 销售订单、采购单、BOM、库存调整等 |
| 适用条件 | 金额范围、部门、角色、客户、供应商等 |
| 审批级次 | 第 1 级、第 2 级等 |
| 审批角色 | 可选 |
| 审批人员 | 可选 |
| 是否必须审批 | 是 / 否 |
| 超时提醒时间 | 超过多久提醒 |
| 状态 | 启用、停用 |
| 备注 | 规则说明 |

审批匹配规则：

- 单据提交审批时，系统根据单据类型和适用条件匹配审批规则。
- 如果匹配到多条规则，需要按优先级或更具体条件选择。
- 如果没有匹配到审批规则，高敏单据不得直接通过，应提醒管理员配置审批规则。
- 审批规则变更只影响后续新提交审批的单据，不影响已在审批中的单据。

审批来源单据关联：

- 审批单需要支持关联销售订单、采购单、进货单、BOM、退货单、库存调整、收付款单等多种来源单据。
- 第一阶段建议采用 Django `GenericForeignKey`，字段包括 `source_content_type`、`source_object_id` 和 `source_doc_type`。
- 为提高审批待办列表和历史审批查询性能，审批单需要冗余保存 `source_no`、`source_title`、`source_summary`。
- `source_no` 保存来源单据号，例如销售订单号、采购单号或 BOM 编号。
- `source_title` 保存短标题，例如“销售订单 SO202606080001”。
- `source_summary` 使用 JSON 保存审批摘要，例如客户 / 供应商、金额、物料数量、交期、申请人、风险提示和主要变更差异。
- 审批提交时生成摘要快照；来源单据后续变更不应静默改变已提交审批的摘要，必要时应撤回后重新提交。
- 审批详情页可以根据权限跳转来源单据，但审批待办列表优先使用审批单自身冗余字段展示，避免大量跨表查询。
- `approvals` 和 `approval_logs` 不直接依赖具体业务 app 的模型逻辑；审批通过后的业务状态变更由来源业务服务执行。

审批简化规则：

- 为适应小公司使用，审批规则需要支持低风险单据免审或简化审批，但必须由管理员显式配置。
- 支持按金额设置小额采购简化审批，例如小额采购需求审核通过后，采购单和进货单可只做确认不再重复走完整审批。
- 支持同一审批人连续出现在多个审批节点时自动跳过重复节点，避免同一个人重复审批同一业务。
- 支持采购需求、采购单、进货单按业务需要合并为一个采购审批链，但进货入库仍必须有验收确认记录。
- 高风险事项不能自动跳过审批，例如大额采购、库存调整、权限变更、敏感导出和收付款确认。
- 所有免审、跳过或合并审批都必须写入审批日志，记录命中的规则和原因。

审批页面体验规则：

- 审批页面需要显示单据摘要，让审批人不用打开多个页面也能判断是否同意。
- 摘要内容应包括申请人、申请时间、客户或供应商、金额、交期、物料、数量、附件和备注。
- 如果是变更审批，需要突出显示变更前和变更后的差异。
- 如果是价格、库存、权限或敏感导出审批，需要显示风险提示。
- 审批页面需要显示历史审批意见和当前审批节点。
- 驳回时必须填写原因；常见原因可以提供快捷选项，例如资料不全、价格异常、数量不符、附件缺失。
- 审批通过后，应回到来源单据并提示下一步动作。

### 12.3 远程审批

审批人员不在公司时，可以支持远程审批。

远程审批方式建议：

| 方式 | 说明 | 适用情况 |
| --- | --- | --- |
| Web 系统审批 | 审批人通过浏览器登录 ERP 审批 | 首选方式 |
| 手机端 H5 审批 | 手机浏览器打开审批页面 | 适合外出审批 |
| 企业微信/钉钉审批提醒 | 消息提醒，点击后进入 ERP 审批 | 后续可扩展 |
| 邮件提醒 | 邮件通知审批事项 | 可作为辅助提醒 |

远程审批安全要求：

- 审批人必须登录账号，不能只靠一个公开链接审批。
- 财务、采购价格、BOM、权限等敏感审批建议增加二次验证。
- 审批链接需要有有效期。
- 审批页面只显示当前审批需要的信息，不暴露无关客户和价格数据。
- 所有远程审批必须记录 IP、设备、审批时间、审批意见。
- 审批通过、驳回、转交都需要留日志。
- 异地或异常设备审批可提醒管理员。

### 12.4 审批动作

审批人可执行：

- 同意。
- 驳回。
- 转交。
- 加签。
- 退回修改。

审批字段建议：

| 字段 | 说明 |
| --- | --- |
| 审批单号 | 系统自动生成 |
| 来源单据 | 销售订单、采购需求、BOM、退货单等 |
| 审批类型 | 订单审批、价格审批、BOM 审批等 |
| 当前审批人 | 当前待处理人员 |
| 审批状态 | 待审批、已通过、已驳回、已转交、已撤回 |
| 审批意见 | 审批人填写 |
| 审批时间 | 操作时间 |
| 审批 IP | 远程审批时记录 |
| 审批设备 | 浏览器、手机、电脑等 |

### 12.5 审批超时与代理

如果审批人员不在公司或无法及时处理，需要支持代理和超时规则。

建议规则：

- 审批超过设定时间后提醒审批人。
- 超过更长时间后提醒管理员或上级主管。
- 审批人可提前设置代理审批人。
- 代理审批必须有授权时间范围。
- 高敏事项不建议自动代理，例如权限变更、管理员密码重置、BOM 启用。
- 审批超时不能默认自动通过。

### 12.6 固定 IP 与远程访问

远程审批不一定必须使用固定公网 IP，但必须有稳定、安全的访问方式。

建议方案：

| 部署方式 | 是否需要固定 IP | 说明 |
| --- | --- | --- |
| 公司内网使用 | 不需要 | 只在公司内部电脑访问，不做外部审批 |
| 公司本地服务器 + 外部远程审批 | 建议需要 | 固定公网 IP + 域名 + HTTPS，访问最稳定 |
| 云服务器部署 | 通常自带固定公网 IP | 推荐小规模正式系统使用，远程审批更方便 |
| 没有固定公网 IP | 可用动态域名或 VPN | 可以实现，但稳定性和维护成本更高 |
| 高安全要求 | 建议 VPN 或零信任访问 | 审批人先连 VPN，再访问 ERP |

安全要求：

- 数据库不能直接暴露到公网。
- 公网只开放 ERP Web 服务，不开放 PostgreSQL 数据库端口。
- 远程访问必须使用 HTTPS。
- 管理后台、数据库管理工具、备份下载不建议直接对公网开放。
- 可限制管理员只能从公司网络或 VPN 登录。
- 远程审批需要记录 IP、设备、时间和审批动作。

推荐结论：

```text
只在公司内用：不需要固定 IP。
需要远程审批：建议云服务器或固定公网 IP + 域名 + HTTPS。
安全要求高：建议 VPN / 零信任访问，不直接暴露管理入口。
```

## 13. 验收标准

### 13.1 基础资料

- 可以新增、编辑、停用客户。
- 可以为客户维护多个产品资料。
- 可以为客户维护多个收件地址和电话。
- 可以新增、编辑、停用供应商。
- 可以新增、编辑、停用物料编码。
- 物料编码支持人工输入英文字母和数字，并校验唯一性。
- 物料档案可以配置库存基础单位和数量精度。
- 系统可以提供建议编码，但允许用户按公司习惯手动修改。
- 可以为库存零部件、原料、包装材料保存默认采购价格和最新采购价格。
- 每一批采购入库都可以保存独立采购单价，并形成库存批次成本。
- 成品需要维护 BOM，BOM 支持版本保存和历史追溯。
- 可以维护物料单位换算规则。
- 可以通过初始化模板导入客户、供应商、物料、BOM 和期初库存。
- 初始化导入需要生成校验结果和确认记录。

### 13.2 销售与库存

- 销售订单审核确认后能自动检查库存。
- 销售订单可以上传销售合同附件，并以销售订单作为业务依据。
- 销售订单支持订单头和多个明细行。
- 销售订单可以从客户产品默认价或历史成交价带出销售价格。
- 销售订单可以快速新增缺失的客户产品、收件地址或发起资料补全任务。
- 销售订单审核后关键字段变更需要记录变更前后内容和变更原因。
- 销售订单明细行可以分别锁定 BOM 编号和 BOM 版本。
- 销售订单确认时能锁定 BOM 编号和 BOM 版本。
- 生产指令单能沿用销售订单锁定的 BOM 编号和版本。
- 库存不足时欠料自动标红。
- 能看到缺少哪些原料、配件或包装材料。
- 部分缺料时，MVP 阶段按整单齐套判断并显示各物料缺口。
- 库存满足生产条件时，第一期可以手动创建生产指令单，第二期支持自动生成生产指令单。
- 客户退货后可以转入成品库存。
- 出库或生产领料时可以追溯使用的是哪一批库存及对应采购成本。
- 可以根据生产指令单创建生产领料单。
- 生产领料可以按 BOM 自动计算建议领料数量，并支持分批领料。
- 生产领料和销售出库可以按先进先出自动推荐批次，并允许有权限人员调整。
- 生产完成后可以创建生产入库单，更新成品库存和生产指令单状态。
- 销售发货可以创建销售出库单，支持按订单明细分批出库。
- 多明细行销售订单可以按明细状态汇总订单头状态。
- 欠料提醒可以生成采购需求，并在采购到货后重新计算齐套状态。
- 可以创建盘点单，录入实盘数量，审核后生成盘点调整流水。
- 盘点单需要保存账面库存快照，并提示盘点期间发生的库存变化。
- 并发出库或领料时不能出现负库存或超扣库存。
- 系统为单一仓库，支持多库位库存和库位移库。
- 系统不要求条码扫码出入库。
- 销售出库单、生产领料单、生产入库单、盘点表和送货单可以打印。
- 仓库打印单据可以按库位排序，便于拣货。

### 13.3 收款与付款

- 可以登记客户收款记录，并上传收款凭证附件。
- MVP 阶段客户收款按销售订单核销，第二期支持按客户对账单核销。
- 可以支持一张销售订单多次收款。
- 客户退货后如果出现超收，可以形成待处理余额。
- 已确认客户收款可以通过红冲更正，并保留原收款、原核销和反向核销记录。
- 客户待处理余额可以转预收款、核销到其他订单、登记退款或关闭。
- 系统能计算客户应收、已收和未收金额。
- 可以登记供应商付款记录，并上传付款凭证附件。
- MVP 阶段供应商付款按进货单核销，第二期支持按供应商对账单核销。
- 可以支持一张进货单多次付款。
- 供应商退货后如果出现超付，可以形成待处理余额。
- 已确认供应商付款可以通过红冲更正，并保留原付款、原核销和反向核销记录。
- 供应商待处理余额可以转预付款、核销到其他进货单、登记退款或关闭。
- 系统能计算供应商应付、已付和未付金额。
- 金额、单价、数量计算不能使用浮点数，并能按规则保留小数。
- 收付款模块不直接连接银行账号。

### 13.4 借样

- 可以创建客户借样单。
- 可以设置借样期限。
- 到期前和逾期后能通过系统消息提醒。
- 可以登记借样归还，支持部分归还和归还入库。
- 借样归还入库后需要更新库存、借样归还数量和逾期提醒状态。
- 借样品种可以转成销售订单。

### 13.5 采购与供应商

- 可以录入进货。
- 可以从采购需求生成采购单。
- 一张采购单可以分多次进货。
- 采购单明细可以追踪采购数量、已到货数量和未到货数量。
- 进货单可以从供应商价格档案或物料最新采购价带出默认单价。
- 合格品可以入库。
- 进货入库需要同时生成库存批次、更新当前库存并写入库存流水。
- 不良品可以退给供应商。
- 可以登记供应商付款记录，但不直接连接银行账号。
- 采购单、进货单和供应商付款相关列表可以按供应商、状态、日期和未到货 / 未付款筛选。

### 13.6 权限

- 管理员可以查看所有模块。
- 销售只能看到自己的客户。
- 采购不能查看客户敏感信息。
- 仓库不能查看销售价格，除非被授权。
- 无权限访问或无权限操作时，系统需要给出明确提示，避免用户误以为数据不存在。

### 13.7 备份与导出

- 系统每天能自动备份。
- 每日备份保存到本机指定目录。
- 备份包含数据库和附件文件，并记录校验值。
- 至少可以在测试环境完成备份恢复验证。
- 关键模块可以导出 Excel。
- 敏感 Excel 导出需要记录导出人、时间、模块和筛选条件。
- Excel 导入需要先校验并生成错误清单，通过确认后再写入数据。
- 管理员可以查看备份记录。
- 用户可以保存常用列表筛选条件。
- 批量导出、批量生成采购需求等操作需要显示成功和失败明细。

### 13.8 审批

- 可以配置不同单据的审批人。
- 可以按单据类型、角色、人员和金额条件配置审批规则。
- 可以支持多级顺序审批。
- 可以配置小额采购、重复审批人等简化审批规则，并记录跳过原因。
- 审批页面可以显示单据摘要、变更差异、风险提示、附件和历史审批意见。
- 审批驳回必须填写原因。
- 审批人不在公司时可以通过 Web 或手机浏览器远程审批。
- 远程审批必须登录账号。
- 敏感审批可以开启二次验证。
- 审批通过、驳回、转交、加签都需要记录日志。
- 审批超时可以提醒或转交，但不能默认自动通过。
- 销售订单审核、退货审核、采购审核必须支持。

### 13.9 开发与上线

- 正式环境配置不能使用开发环境数据库、密钥或调试模式。
- 数据库结构变更需要通过迁移脚本执行。
- 每次发布前需要完成数据库和附件备份。
- 上线前需要通过核心端到端流程测试。
- 系统支持查看版本发布记录。
- 附件访问需要经过权限校验，不能直接暴露本机文件目录。
- 首页工作台可以显示待审批、欠料、借样逾期、待发货、待入库和我的待处理任务。
- 已停用用户不能登录，但历史单据和日志仍显示原姓名并标注已停用。
- 第一阶段以桌面浏览器为主，审批页面适配手机浏览器。
- 断网或保存失败时，系统需要明确提示，不能让用户误以为操作成功。

## 14. 已确认业务规则

以下业务规则已确认，后续开发按此执行：

1. 物料编码公司没有现成规则，系统支持人工输入英文字母和数字，并校验唯一性。
2. 成品需要 BOM 管理，BOM 需要支持版本保存和历史追溯。
3. 销售合同作为销售订单附件保存，销售订单作为销售、库存、生产、发货和对账依据。
4. 销售订单审核、退货审核、采购审核都需要支持。
5. 对账需要同时支持按月、按订单、按客户自定义周期。
6. 借样超期提醒通过系统消息提醒。
7. 每日备份保存到本机。
8. 系统采用单一仓库，但需要支持多库位。
9. 不需要条码扫码出入库。
10. 不需要开票模块。
11. 需要支持客户收款和供应商付款模块，但不直接连接银行账号。
12. 第一阶段采用 Django 单体 Web 应用、PostgreSQL、服务端 Session。
13. MVP 阶段二次验证采用再次输入当前登录密码。
14. MVP 阶段收付款按销售订单和进货单核销，第二期再支持按对账单核销。
15. 第一阶段手动创建生产指令单，第二期再支持自动生成生产指令单。
16. 库存扣减、进货入库、收付款核销必须使用数据库事务和并发控制。
17. 备份需要包含数据库和附件文件，并进行校验和恢复演练。
18. 金额、数量、单位换算必须使用 decimal / numeric，不使用浮点数。
19. 附件、Excel 导入导出、系统配置、发布记录和测试流程需要纳入开发范围。
20. 销售订单采用订单头和多明细行结构，库存检查、BOM 锁定和生产指令按明细行处理。
21. 采购流程采用采购需求 -> 采购单 -> 进货单，支持分批到货。
22. 销售订单头状态由明细行状态汇总，明细行保留各自库存检查状态和业务状态。
23. 销售订单允许先保存草稿，提交审核前再校验客户产品、成品编码和 BOM 等前置资料。
24. 生产领料、生产入库、销售出库和盘点必须作为独立业务单据纳入库存模块。
25. 欠料提醒可以生成采购需求，并在采购到货后重新计算齐套状态。
26. 生产领料和销售出库默认按先进先出自动推荐库存批次，允许有权限人员调整并记录原因。
27. 审批规则需要支持按单据类型、角色、人员、金额和多级顺序配置。
28. 审批规则需要支持小额采购、重复审批人等简化审批配置，但高风险事项不能自动跳过。
29. 首页工作台需要显示待审批、欠料、借样逾期、待发货、待入库和我的待处理任务。
30. 已停用用户不能登录，但历史单据和日志仍显示原姓名并标注已停用。
31. 上线前需要支持初始化导入客户、供应商、物料、BOM、期初库存、未完成订单和期初应收应付。
32. 销售订单审核后关键字段变更需要走变更记录或变更审批，不能静默修改。
33. 销售订单需要支持销售价格自动带出、缺失资料快速补充和历史订单复制。
34. 核心列表需要支持搜索、筛选、常用筛选保存、批量操作和无权限提示。
35. 第一阶段不做条码扫码时，需要支持生产领料单、生产入库单、销售出库单、送货单和盘点表打印。
36. 系统消息需要分级，并提供来源单据、负责人、建议动作和一键跳转。
37. 审批页面需要显示单据摘要、变更差异、风险提示、附件和历史审批意见。
38. 第一阶段桌面浏览器优先，手机浏览器主要支持审批；断网或保存失败必须明确提示。
39. 库存齐套检查必须按“理论需求 -> 损耗 -> 单位换算 -> 物料精度 -> 可用库存比较”的顺序计算。
40. BOM 必需子件缺料会阻止齐套；可选子件缺料只提示警告，不阻止生产。
41. 单号生成第一阶段采用 `document_sequences` 表加行级锁，不采用独立 PostgreSQL SEQUENCE 作为业务单号主方案。
42. 采购入库事务内只完成入库和单据状态更新，欠料重检和消息通知在事务提交后执行。
43. 金额、数量、BOM 用量、损耗率、单位换算比例需要按文档建议的 `numeric` 精度建表。
44. Django Session 第一阶段使用数据库 Session，支持管理员强制注销会话。
45. 定时任务第一阶段使用 Django 管理命令加计划任务，并通过 `background_jobs` 防止重复运行。
46. 库存批次自动推荐按 `received_at ASC, batch_no ASC` 先进先出，冻结、作废和已用完批次不得被推荐。
47. 第一阶段采用单体部署架构，但正式环境需要明确 Web 服务、数据库、附件目录、备份目录、日志目录和定时任务的运行位置。
48. 库存、审批、财务、附件、消息和系统配置需要通过各自服务模块写入，其他模块不能直接修改核心表。
49. 后台任务需要支持防重复、失败重试、幂等处理和管理员可见的失败提醒。
50. 正式环境需要健康检查、错误日志、后台任务日志、备份日志、磁盘空间告警和慢请求 / 慢查询记录。
51. 借样归还需要作为独立归还单据处理，支持部分归还、归还入库、逾期状态更新和库存流水。
52. 已确认收款和付款的更正采用红冲或反向调整，不能直接删除原记录和原核销明细。
53. 客户预收 / 超收余额和供应商预付 / 超付余额需要支持转余额、核销其他单据、登记退款和关闭，并保留处理流水。

## 16. 工程落地规格

以下内容用于把业务需求转为可开发、可测试、可上线的工程规格。正式开发时，程序员应优先按本章细化数据库模型、服务方法、页面动作和测试用例。

### 16.1 命名与编码约定

数据库和代码建议统一使用英文编码，前端展示中文名称。

命名规则：

- 数据库表名使用小写复数形式，例如 `sales_orders`、`sales_order_items`。
- 数据库字段使用小写下划线，例如 `created_at`、`material_code`。
- 状态字段保存英文枚举码，前端显示中文。
- 金额字段后缀建议使用 `_amount`，单价字段后缀建议使用 `_price`。
- 数量字段后缀建议使用 `_qty`，日期字段后缀建议使用 `_date`。
- 外键字段后缀建议使用 `_id`，业务单号字段后缀建议使用 `_no`。

状态编码示例：

| 中文状态 | 建议编码 |
| --- | --- |
| 草稿 | draft |
| 待审核 | pending_approval |
| 已驳回 | rejected |
| 已确认 | confirmed |
| 待 BOM 处理 | pending_bom |
| 生产中 | in_production |
| 已发货 | shipped |
| 已完成 | completed |
| 已作废 | voided |

开发注意：

- 不建议在数据库中直接保存中文状态，避免后续改文案影响业务判断。
- 状态枚举需要集中定义，不能在多个页面手写字符串。
- 所有状态流转必须通过服务方法执行，不允许页面直接修改状态字段。

| customer_credit_balance_transactions | `transaction_no` |
| supplier_credit_balance_transactions | `transaction_no` |
| approvals | `approval_no` |
| document_sequences | `prefix + sequence_date` |
| pending_events | `idempotency_key` |

约束说明：

- BOM 被驳回或作废后，允许同一成品同一版本重新提交新记录；唯一约束只限制草稿、待审核、已启用、已停用等活跃或历史有效记录。
- 如果数据库支持部分唯一索引，建议对 `boms(finished_material_id, bom_version)` 增加条件唯一索引：`WHERE status NOT IN ('rejected', 'voided')`。
- 销售订单明细默认不允许同一订单下重复选择同一客户产品；如确实需要拆成交期、价格或包装不同的多行，应增加行分组或拆分原因字段，并由业务规则放开。
- 红冲表需要限制同一原始收付款单的同一幂等请求不能重复生成红冲记录，可增加 `source_receipt_id + idempotency_key` 或 `source_payment_id + idempotency_key` 唯一约束。
- 待处理余额使用流水需要限制同一余额记录的同一幂等请求不能重复扣减，可增加 `credit_balance_id + idempotency_key` 唯一约束。
- `pending_events.idempotency_key` 用于防止同一业务事件重复入队，例如 `purchase_received:123`。

索引建议：

| 场景 | 建议索引 |
| --- | --- |
| 销售订单列表 | `customer_id`、`status`、`created_at`、`delivery_date` |
| 销售订单明细查询 | `sales_order_id`、`material_id`、`line_status` |
| 库存查询 | `material_id`、`location_id`、`inventory_type` |
| 批次先进先出 | `material_id + location_id + received_at + batch_no` |
| 欠料列表 | `status`、`material_id`、`sales_order_item_id` |
| 审批待办 | `current_approver_id`、`status`、`created_at` |
| 收付款核销 | `customer_id` / `supplier_id`、`status`、`payment_date` |
| 借样归还 | `sample_loan_id`、`customer_id`、`status`、`return_date` |
| 客户待处理余额 | `customer_id`、`status`、`remaining_amount` |
| 供应商待处理余额 | `supplier_id`、`status`、`remaining_amount` |
| 事务后事件 | `status`、`event_type`、`next_retry_at` |
| 系统消息 | `receiver_id`、`status`、`level`、`created_at` |

### 16.4 状态机动作规格

每个核心单据需要定义“状态 + 动作 + 权限 + 后置影响”。开发时不能只写状态名称。

状态机字段建议：

| 项目 | 说明 |
| --- | --- |
| 当前状态 | 操作前状态 |
| 动作 | 用户点击的按钮或系统触发动作 |
| 所需权限 | 角色或权限码 |
| 校验条件 | 必填字段、库存、金额、状态、审批规则等 |
| 目标状态 | 操作后状态 |
| 写入数据 | 需要新增或更新哪些表 |
| 事务要求 | 是否必须事务、是否需要锁 |
| 消息通知 | 是否生成系统消息 |
| 日志 | 是否写审批日志、审计日志或库存流水 |

销售订单关键动作示例：

| 当前状态 | 动作 | 目标状态 | 关键校验 |
| --- | --- | --- | --- |
| draft | 提交审核 | pending_approval | 客户、明细行、数量、价格必填 |
| pending_approval | 审核通过 | confirmed / pending_bom | 审批人权限、BOM 和库存检查 |
| pending_approval | 驳回 | rejected | 必须填写驳回原因 |
| confirmed | 生成生产指令单 | in_production | 成品不足且物料齐套 |
| confirmed | 创建销售出库单 | shipped / confirmed | 成品库存充足，出库数量不超销售数量 |
| confirmed / in_production | 订单变更 | 原状态或待审核 | 变更字段、影响范围、审批规则 |
| shipped | 完成订单 | completed | 所有明细已发货，收付款或对账条件满足 |
| draft / rejected | 作废 | voided | 未产生库存或财务影响 |

库存相关动作必须额外定义锁定对象：

- 销售出库：锁定 `inventory_batches` 和 `inventory`。
- 生产领料：锁定子件对应的 `inventory_batches` 和 `inventory`。
- 采购入库：锁定或新建 `inventory_batches`，更新 `inventory`。
- 盘点调整：锁定盘点范围内的 `inventory` 和相关批次。

### 16.5 服务方法与事务边界

后端建议把关键业务写成服务方法，页面只负责收集输入和展示结果。

核心服务方法建议：

| 服务方法 | 主要职责 |
| --- | --- |
| `confirm_sales_order` | 审核通过、库存检查、BOM 锁定、欠料生成 |
| `recheck_sales_order_inventory` | BOM 补齐或采购入库后重新检查库存 |
| `create_purchase_request_from_shortages` | 从欠料提醒生成采购需求 |
| `confirm_purchase_receipt` | 验收、生成批次、更新库存、写库存流水 |
| `confirm_material_requisition` | 生产领料扣库存、写流水、更新生产状态 |
| `confirm_production_receipt` | 成品入库、写批次、更新生产指令和销售行 |
| `confirm_sales_shipment` | 销售出库扣库存、写流水、更新销售状态 |
| `confirm_stock_count_adjustment` | 盘点调整库存和流水 |
| `confirm_sample_return` | 借样归还入库、写库存流水、更新借样归还数量和逾期状态 |
| `confirm_customer_receipt` | 确认收款和核销 |
| `reverse_customer_receipt` | 对已确认客户收款进行红冲，生成反向核销并重算订单收款状态 |
| `apply_customer_credit_balance` | 处理客户预收、超收、退款或转核销余额 |
| `confirm_supplier_payment` | 确认付款和核销 |
| `reverse_supplier_payment` | 对已确认供应商付款进行红冲，生成反向核销并重算进货单付款状态 |
| `apply_supplier_credit_balance` | 处理供应商预付、超付、退款或转核销余额 |
| `apply_approval_action` | 审批通过、驳回、转交、加签 |
| `void_document` | 作废单据并校验是否可作废 |
| `process_pending_events` | 处理事务提交后的持久化业务事件，例如欠料重检、消息通知和汇总校验 |

事务边界规则：

- 上述服务方法如果涉及库存、收付款、审批状态或单号生成，必须在数据库事务内执行。
- 服务方法需要先做权限和状态校验，再锁定需要修改的数据。
- 事务中不建议执行耗时文件处理或 Excel 导出，应先提交业务事务，再触发后台任务。
- 每个服务方法需要返回明确结果，例如成功、失败原因、生成的单号、下一步动作。

### 16.8 URL、接口与请求规范

第一期采用 Django 单体应用时，也需要统一页面路由和动作接口，避免后续维护混乱。

路由建议：

| 类型 | 示例 | 说明 |
| --- | --- | --- |
| 列表页 | `/sales/orders/` | 查询和筛选 |
| 新增页 | `/sales/orders/new/` | 新建草稿 |
| 详情页 | `/sales/orders/{id}/` | 查看详情 |
| 编辑页 | `/sales/orders/{id}/edit/` | 编辑草稿或允许编辑字段 |
| 动作接口 | `/sales/orders/{id}/submit/` | 提交、审核、确认等动作 |
| 弹窗查询 | `/api/materials/search/` | 物料、客户、供应商等模糊搜索 |

请求规则：

- 查询和页面展示使用 `GET`。
- 新增、修改、提交审核、确认出入库、确认收付款、作废等动作必须使用 `POST`。
- 任何会改变数据的动作不能用 `GET`。
- 所有 `POST` 请求需要 CSRF 防护。
- 关键动作需要后端再次校验权限、状态和业务条件。
- 确认出库、确认入库、确认核销、生成采购需求等动作建议使用幂等令牌，避免重复提交。
- 页面动作成功后建议使用 PRG 模式，即 `POST -> Redirect -> GET`，避免浏览器刷新重复提交。
- 搜索、自动补全、批次推荐等局部交互可以使用 JSON 接口。

接口返回建议：

| 场景 | 返回内容 |
| --- | --- |
| 页面表单提交成功 | 跳转到详情页或列表页，并显示成功消息 |
| 页面表单提交失败 | 返回原页面，显示字段错误和业务错误 |
| JSON 动作成功 | `success`、生成单号、下一步动作 |
| JSON 动作失败 | `error_code`、中文提示、建议处理动作 |

### 16.9 Django 代码结构建议

代码结构需要按业务模块组织，便于多人维护。

建议应用拆分：

| Django app | 主要内容 |
| --- | --- |
| `accounts` | 用户、角色、权限、登录、安全 |
| `masterdata` | 客户、供应商、物料、单位换算 |
| `bom` | BOM 主表、明细、版本和审批 |
| `sales` | 销售订单、客户退货、借样、销售出库 |
| `purchase` | 采购需求、采购单、进货、供应商退货 |
| `inventory` | 库存、批次、库位、流水、盘点 |
| `production` | 生产指令、领料、生产入库 |
| `finance` | 收款、付款、核销、对账 |
| `approvals` | 审批规则、审批单、审批日志 |
| `notifications` | 系统消息和提醒 |
| `files` | 附件、导入、导出、打印 |
| `system` | 系统配置、备份、后台任务、发布记录 |

每个业务 app 建议结构：

```text
models.py      数据模型
forms.py       表单和字段校验
views.py       页面视图和动作入口
services.py    业务服务和事务逻辑
selectors.py   查询封装和列表筛选
permissions.py 权限判断
urls.py        路由
tests.py       测试
```

开发规则：

- `views.py` 不直接写复杂业务事务，复杂逻辑放到 `services.py`。
- 列表查询和权限过滤放到 `selectors.py` 或查询服务中。
- 权限判断集中封装，页面、导出、打印和 JSON 接口都复用。
- 关键服务方法需要有单元测试或集成测试。

模块依赖边界：

- 业务 app 依赖方向需要从基础到业务单向流动，禁止形成循环 import。
- `inventory` 是库存批次、当前库存和库存流水的唯一写入入口；销售、采购、生产、借样、退货、盘点都必须调用 `inventory.services` 完成库存增减，不能直接更新库存表。
- `sales` 负责销售订单、销售明细、客户退货和借样业务状态；涉及出库扣减时调用库存服务，涉及收款余额时只读取财务服务结果。
- `purchase` 负责采购需求、采购单、进货和供应商退货；涉及入库、退货出库和批次成本时调用库存服务。
- `production` 负责生产指令、领料单和生产入库单；领料扣减和成品入库都通过库存服务执行。
- `finance` 负责收款、付款、核销和对账余额计算；不能直接修改销售订单、采购单或进货单的原始金额，只能写核销明细和财务状态。
- `approvals` 只负责审批规则、审批实例和审批日志；审批通过后由对应业务服务执行状态变更，审批模块不直接修改业务单据核心状态。
- `notifications` 只负责系统消息生成、读取和状态变更；业务模块可以调用消息服务生成提醒，消息模块不反向依赖具体业务服务。
- `files` 负责附件元数据、上传、下载、导入、导出和打印文件；业务模块只保存附件关联关系，不直接拼接本机文件路径。
- `system` 负责配置、备份、后台任务、发布记录和健康检查；业务模块读取配置应通过统一配置服务。

建议依赖层级：

| 层级 | App | 依赖规则 |
| --- | --- | --- |
| 基础层 | `accounts`、`masterdata`、`system` | 尽量不依赖具体业务 app |
| 资源层 | `inventory`、`bom`、`files` | 依赖基础层，提供库存、BOM、附件能力 |
| 业务层 | `sales`、`purchase`、`production` | 依赖基础层和资源层，执行业务单据流程 |
| 财务层 | `finance` | 依赖销售、采购和核销对象，但不修改业务单据原始金额 |
| 横切层 | `approvals`、`notifications` | 通过通用关联、事件或服务接口协作，不直接耦合具体业务状态机 |

横切模块解耦规则：

- `approvals` 采用通用来源单据关联和摘要快照，不为每种业务单据写一套审批表。
- `notifications` 保存来源单据类型、来源单据 ID、来源单号和跳转链接，不直接 import 各业务模块服务。
- `pending_events` 用于承接事务后副作用，事件处理器可以调用业务服务，但业务事务内不应反向等待事件处理完成。
- `audit_logs` 保存通用操作信息和对象标识，不要求对每种业务表建立硬外键。

跨模块调用规则：

- 跨模块写操作优先通过服务方法调用，不允许在其他模块页面或模型信号中直接改核心表。
- 不建议使用 Django model `save()` 信号承载复杂业务流程，库存、审批、核销和单号生成必须显式调用服务方法，方便测试和事务控制。
- 服务方法内部可以调用其他模块服务，但需要明确事务边界和锁定顺序，避免循环调用和死锁。
- 如果两个模块互相需要结果，应抽出更底层的领域服务或查询服务，不要形成循环 import。
- 对外展示和导出使用 `selectors.py` 或查询服务，避免页面散落复杂 ORM 查询。

数据库迁移与初始化规则：

- 所有表结构变更必须通过 Django migration，不允许正式环境手工改表后不提交迁移文件。
- 初始角色、权限码、系统配置、状态枚举、默认库位和单号前缀建议通过数据迁移或初始化管理命令写入。
- 权限码、状态编码、单号前缀一旦上线，不应随意改名；确需变更时必须提供数据迁移和兼容说明。
- 每次发布前应在测试环境执行迁移，并确认迁移可以重复执行、可以在已有数据上运行。
- 初始化导入脚本需要可追溯，记录导入文件、导入人、导入时间、成功数量、失败数量和错误明细。

## 17. 后台任务、错误处理与模板规格

### 17.1 后台任务清单

系统需要将耗时、定时和可重试操作纳入后台任务管理。

后台任务建议：

| 任务 | 触发方式 | 建议频率 / 时机 | 失败处理 |
| --- | --- | --- | --- |
| 每日备份 | 定时 | 每天凌晨 | 记录失败并通知管理员 |
| 备份校验 | 定时 | 每天备份后 | 校验失败通知管理员 |
| 借样到期提醒 | 定时 | 每天上午 | 生成系统消息 |
| 借样逾期提醒 | 定时 | 每天上午 | 生成高亮系统消息 |
| 审批超时提醒 | 定时 | 每小时或每天 | 提醒审批人和管理员 |
| 采购到货延期提醒 | 定时 | 每天 | 提醒采购和相关负责人 |
| 库存汇总校验 | 定时 | 每天或每周 | 发现批次汇总不一致时通知管理员 |
| Excel 导出 | 手动触发后台执行 | 大数据量导出时 | 失败生成错误记录 |
| Excel 导入 | 手动触发后台执行 | 用户确认导入后 | 失败生成错误清单 |
| 初始化导入 | 手动触发后台执行 | 上线前 | 失败禁止写入正式数据或回滚 |
| 系统消息清理 | 定时 | 每月 | 归档已处理旧消息 |

任务记录字段建议：

| 字段 | 说明 |
| --- | --- |
| 任务编号 | 系统生成 |
| 任务类型 | backup、export、import、reminder 等 |
| 触发方式 | 定时、手动、系统事件 |
| 发起人 | 手动任务的发起用户 |
| 开始时间 | 任务开始时间 |
| 结束时间 | 任务结束时间 |
| 状态 | pending、running、success、failed、cancelled |
| 输入参数 | JSON 保存筛选条件或模板信息 |
| 结果摘要 | 成功数量、失败数量、文件路径等 |
| 错误信息 | 失败时记录 |

实现建议：

- Django 第一阶段可使用管理命令 + Windows 任务计划程序或 Linux cron。
- 如果后续任务量增加，可引入 Celery 或 Django-Q。
- 后台任务需要避免重复执行，例如同一时间只能有一个每日备份任务。
- 任务失败不能静默，需要在系统消息或后台任务列表中可见。

Django 管理命令建议：

| 命令 | 用途 |
| --- | --- |
| `python manage.py backup_daily` | 执行每日数据库和附件备份 |
| `python manage.py verify_backups` | 校验备份文件可读性和校验值 |
| `python manage.py send_reminders` | 生成借样到期、逾期、采购延期等提醒 |
| `python manage.py check_approval_timeouts` | 检查审批超时并提醒 |
| `python manage.py validate_inventory_summary` | 校验 `inventory` 与 `inventory_batches` 汇总一致性 |
| `python manage.py validate_business_summaries` | 校验已发货、已到货、已入库、收付款余额等业务汇总字段 |
| `python manage.py process_pending_events` | 处理事务提交后待处理业务事件 |
| `python manage.py process_import_jobs` | 处理待执行导入任务 |
| `python manage.py process_export_jobs` | 处理大数据量导出任务 |

后台任务防重复规则：

- 每个管理命令执行前，先在 `background_jobs` 中创建或锁定运行记录。
- 同一任务类型如果已有 `running` 且未超时的记录，新任务不得重复启动。
- 可使用 `SELECT ... FOR UPDATE` 锁定任务记录，或使用唯一约束限制同一任务类型同一时间只能运行一个。
- 任务正常结束后状态改为 `success`；异常时状态改为 `failed` 并记录错误信息。
- 如果任务进程异常退出，超过配置时间仍为 `running` 的任务可标记为 `failed` 后允许重跑。

可靠任务与事务提交规则：

- 库存入库后欠料重检、系统消息、导出文件生成、备份校验等动作不应只依赖页面请求临时执行。
- 关键业务事务提交前，可以在同一事务内写入 `background_jobs` 或待处理事件记录；事务提交后再由后台命令读取并执行。
- 如果业务事务回滚，对应后台任务记录也必须一起回滚，避免处理不存在或未生效的业务数据。
- 任务处理需要支持重试次数、下次重试时间和失败原因。
- 任务执行成功后需要记录处理结果，例如生成的消息数量、更新的欠料数量、导出文件路径或备份校验值。
- 同一业务对象的任务需要支持幂等处理，例如同一进货单重复触发欠料重检时，不能重复生成多条相同的“已齐套”消息。
- 第一阶段可以继续使用管理命令轮询 `background_jobs`；后续如果任务量明显增加，再迁移到 Celery、Redis 或 Django-Q。

### 17.2 错误码与异常提示

用户操作失败时，需要返回清晰、稳定、可测试的错误码和中文提示。

错误码分类建议：

| 错误码前缀 | 类型 | 示例 |
| --- | --- | --- |
| AUTH | 登录、权限、二次验证 | AUTH_NO_PERMISSION |
| STATE | 状态不允许 | STATE_INVALID_TRANSITION |
| STOCK | 库存、批次、库位 | STOCK_NOT_ENOUGH |
| BOM | BOM 和物料需求 | BOM_NOT_ENABLED |
| APPROVAL | 审批规则和审批动作 | APPROVAL_RULE_MISSING |
| PAYMENT | 收付款和核销 | PAYMENT_ALLOCATION_OVER |
| IMPORT | Excel 导入 | IMPORT_REQUIRED_FIELD_MISSING |
| FILE | 附件和打印文件 | FILE_TYPE_NOT_ALLOWED |
| SYSTEM | 系统配置和后台任务 | SYSTEM_CONFIG_MISSING |

常见错误码建议：

| 错误码 | 中文提示 |
| --- | --- |
| AUTH_NO_PERMISSION | 当前账号没有执行该操作的权限 |
| AUTH_SECOND_VERIFY_REQUIRED | 该操作需要二次验证 |
| STATE_INVALID_TRANSITION | 当前单据状态不允许执行该操作 |
| STATE_ALREADY_PROCESSED | 该单据已经处理，请刷新后查看 |
| STOCK_NOT_ENOUGH | 库存不足，无法出库或领料 |
| STOCK_BATCH_LOCKED | 库存批次已冻结，不能使用 |
| STOCK_NEGATIVE_NOT_ALLOWED | 库存不能扣减为负数 |
| BOM_NOT_ENABLED | 成品没有启用 BOM |
| BOM_VERSION_CHANGED | BOM 已发生变化，请重新检查库存 |
| BOM_UNIT_CONVERSION_MISSING | BOM 用量单位无法换算到库存基础单位 |
| APPROVAL_RULE_MISSING | 未配置审批规则，不能提交审批 |
| APPROVAL_COMMENT_REQUIRED | 驳回或退回修改必须填写原因 |
| PAYMENT_ALLOCATION_OVER | 核销金额超过可核销余额 |
| PAYMENT_REVERSAL_OVER | 红冲金额超过可红冲金额 |

| SAMPLE_RETURN_QTY_OVER | 借样归还数量超过可归还数量 |
| IMPORT_REQUIRED_FIELD_MISSING | 导入文件存在必填字段为空 |
| IMPORT_DUPLICATE_KEY | 导入文件存在重复编码或重复单号 |
| FILE_TYPE_NOT_ALLOWED | 附件类型不允许上传 |
| SYSTEM_BACKUP_FAILED | 备份失败，请管理员检查 |

异常处理规则：

- 后端服务方法需要返回错误码和中文提示，不只返回技术异常。
- 前端展示错误时需要说明用户下一步能做什么。
- 并发导致失败时，需要提示用户刷新后重试。
- 系统内部异常需要写入错误日志，但不能向普通用户暴露数据库 SQL、服务器路径或密钥信息。

### 17.3 导入模板规格

导入模板需要固定列名、必填规则和校验规则。

通用导入规则：

- 每个导入模板需要有模板名称和模板版本。
- 面向用户下载的 CSV / Excel 模板首行必须使用中文列名，例如“物料编码”“客户编号”“订单日期”，不能要求普通用户填写英文数据库字段名。
- 后端导入解析需要把中文列名映射到内部英文字段；内部字段名和状态枚举仍使用英文编码，前端和模板显示中文。
- 第一阶段不需要兼容旧英文表头模板；如果上传旧英文表头，应明确提示“请使用系统下载的中文导入模板”。
- 导入前先校验，不通过不写入正式数据。
- 错误清单需要包含行号、字段名、错误原因和建议处理方式。
- 导入成功后需要记录导入任务和来源文件。
- 初始化导入和日常导入需要区分模板。
- 用户填写的数据可以中英文混合；数据库、程序和导入文件必须统一使用 UTF-8 处理，不能因中文客户名、英文型号或特殊字符混合而丢失数据。

优先模板清单：

| 模板 | 用途 | 关键列 |
| --- | --- | --- |
| 物料导入 | 建立物料档案 | 物料编码、名称、类别、单位、数量精度、状态 |
| 客户导入 | 建立客户档案 | 客户编号、客户名称、业务员、结算方式 |
| 客户产品导入 | 建立客户产品资料 | 客户编号、客户产品编号、成品编码、标签要求、包装要求 |
| 供应商导入 | 建立供应商档案 | 供应商编号、供应商名称、联系人、付款方式 |
| BOM 导入 | 建立 BOM 和明细 | 成品编码、BOM 版本、子件编码、用量、单位、损耗率 |
| 期初库存导入 | 上线期初库存 | 物料编码、库位、库存类型、批次号、数量、成本单价 |
| 未收款订单导入 | 上线应收余额 | 销售单号、客户编号、订单金额、已收金额、未收金额 |
| 未付款进货导入 | 上线应付余额 | 进货单号、供应商编号、应付金额、已付金额、未付金额 |
| 未归还借样导入 | 上线借样余额 | 借样单号、客户编号、成品编码、借出数量、已归还数量、已转销售数量、预计归还日期 |
| 预收预付余额导入 | 上线财务余额 | 往来单位编号、余额类型、来源说明、余额金额、已使用金额 |

期初库存导入特别校验：

- 物料编码必须存在。
- 库位必须存在。
- 数量不能为负。
- 成本单价不能为负。
- 批次号不能重复。
- 导入后 `inventory` 汇总必须等于 `inventory_batches` 汇总。

## 18. 测试用例清单

### 18.1 核心业务测试

销售订单测试：

| 用例 | 预期结果 |
| --- | --- |
| 草稿订单缺少成品编码仍可保存 | 保存成功，列表显示缺失项 |
| 缺少成品编码提交审核 | 提交失败并提示缺失明细行 |
| 审核通过且成品库存充足 | 明细库存状态为库存充足，可创建销售出库 |
| 审核通过但 BOM 未启用 | 订单进入待 BOM 处理 |
| BOM 必需子件库存不足 | 明细状态为欠料并生成欠料提醒 |
| BOM 可选子件库存不足 | 明细仍可齐套，但显示可选子件缺料警告 |
| BOM 单位与库存基础单位不同 | 按理论需求、损耗、单位换算、物料精度顺序计算 |
| 找不到单位换算规则 | 库存检查失败并提示维护单位换算 |
| 审核后修改数量 | 需要生成变更记录或走变更审批 |
| 多明细行状态不同 | 订单头按汇总优先级显示 |

采购与库存测试：

| 用例 | 预期结果 |
| --- | --- |
| 欠料生成采购需求 | 欠料状态变为已生成采购需求 |
| 欠料生成采购需求后采购到货入库 | 欠料状态更新为已齐套，系统消息通知可以创建生产指令单 |
| 采购单分批到货 | 采购单明细显示部分到货和未到货数量 |
| 进货确认入库 | 生成库存批次、更新当前库存、写库存流水 |
| 进货确认后相关欠料齐套 | 入库事务提交后触发欠料重检并生成可生产消息 |
| 两人同时扣减同一批库存 | 后提交事务重新校验，不能负库存 |
| 生产领料自动推荐批次 | 按先进先出推荐批次 |
| 冻结批次存在库存 | 自动推荐时跳过冻结批次 |
| 销售出库确认 | 扣减成品库存，更新销售明细发货数量 |
| 盘点期间发生出入库 | 盘点单提示库存变化 |

借样测试：

| 用例 | 预期结果 |
| --- | --- |
| 借样出库 | 扣减成品库存，借样单状态为已出库 |
| 借样部分归还 | 增加库存，更新已归还数量，未归还数量继续保留 |
| 借样归还数量超过未归还数量 | 操作失败并提示超出可归还数量 |
| 逾期借样全部归还 | 借样状态更新，逾期提醒关闭但历史记录保留 |
| 借样部分转销售后再归还 | 只能归还未归还且未转销售数量 |

收付款测试：

| 用例 | 预期结果 |
| --- | --- |
| 一笔收款核销多张订单 | 核销成功并更新订单已收金额 |
| 两笔收款同时核销同一订单 | 不允许超额核销 |
| 客户退货后已核销金额超额 | 生成待处理超收余额 |
| 已确认收款红冲 | 生成红冲记录和反向核销，订单已收金额重新计算 |
| 客户待处理余额核销其他订单 | 锁定余额和目标订单，余额不能重复使用 |
| 付款核销超过进货可付金额 | 操作失败并提示超额 |
| 已确认付款红冲 | 生成红冲记录和反向核销，进货单已付金额重新计算 |
| 供应商待处理余额登记退款 | 生成余额处理流水并更新剩余余额 |

审批测试：

| 用例 | 预期结果 |
| --- | --- |
| 无审批规则提交高敏单据 | 提交失败并提示配置审批规则 |
| 同一审批人重复节点 | 按规则自动跳过重复节点并记录原因 |
| 驳回不填原因 | 操作失败 |
| 手机浏览器审批 | 能查看摘要、附件并提交审批动作 |

### 18.2 权限与安全测试

权限测试：

| 用例 | 预期结果 |
| --- | --- |
| 销售查看其他销售客户 | 无权限或不可见 |
| 销售角色只配置销售权限后登录 | 主导航只显示销售相关入口，直接访问采购、库存、生产、财务 URL 返回无权限 |
| 采购角色查看欠料提醒 | 可以查看欠料提醒并按权限生成采购需求 |
| 老板只读角色查看多模块 | 可以进入授权模块列表和详情，不能执行处理动作，不能进入权限管理 |
| 仓库查看销售价格 | 默认不可见 |
| 采购查看客户电话 | 默认不可见 |
| 无导出权限点击导出 | 操作失败并提示无权限 |
| 停用用户登录 | 登录失败 |
| 密码修改后旧 Session | 旧 Session 失效 |
| 权限清单页面 | 普通管理员界面优先显示中文权限名称，不以英文权限码作为主要可见文案 |

安全测试：

- 上传不允许的附件类型，应拒绝上传。
- Excel 导出以 `=` 开头的内容，应防止公式注入。
- 普通用户不能直接访问附件文件路径。
- 敏感操作需要二次验证。
- 备份文件不能被普通业务账号下载。

## 19. 开发交付与迭代管理

### 19.1 开发拆分原则

第一期开发建议按可验收闭环拆分，不按技术层孤立拆分。

建议迭代：

| 迭代 | 交付内容 |
| --- | --- |
| 迭代 1 | 登录、用户、角色、权限、基础配置 |
| 迭代 2 | 客户、供应商、物料、单位换算、附件 |
| 迭代 3 | BOM、BOM 导入、BOM 审批 |
| 迭代 4 | 库存、库位、批次、库存流水 |
| 迭代 5 | 销售订单、库存检查、欠料提醒 |
| 迭代 6 | 采购需求、采购单、进货入库 |
| 迭代 7 | 生产指令、生产领料、生产入库 |
| 迭代 8 | 销售出库、客户退货、借样 |
| 迭代 9 | 收款、付款、核销 |
| 迭代 10 | 审批规则、工作台、系统消息 |
| 迭代 11 | 导入导出、打印、初始化导入 |
| 迭代 12 | 备份、测试、上线演练 |

### 19.2 每个功能交付标准

每个功能上线前至少满足：

- 数据表和迁移脚本完成。
- 页面列表、详情、新增、编辑动作完成。
- 权限校验完成。
- 关键服务方法有事务保护。
- 必要日志和系统消息完成。
- 导入、导出或打印需求完成。
- 单元测试或集成测试覆盖核心成功和失败场景。
- 验收标准中对应条目通过。

### 19.3 代码实现注意事项

开发注意：

- 业务逻辑优先放在服务层，不要散落在多个视图函数中。
- 库存扣减、收付款核销、审批状态变更必须统一入口。
- 不要让前端传入金额汇总后直接相信，后端需要重新计算。
- 不要让前端传入库存检查结果后直接扣库存，扣减前必须重新查库存。
- 文件路径、备份目录、数据库密码不能写死。
- 导出和打印必须重新检查权限。
- 页面显示的中文文案可以调整，但状态编码和权限编码不能随意变更。

- 每张表的 Django Model 草案。

- 每个服务方法的输入、输出和异常码。

- 每个打印模板的样式样稿。
