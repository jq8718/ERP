# 生产上线部署与运维手册

生成时间：2026-06-11

本文档用于把当前 Django ERP 项目从开发环境切换到生产环境，并给出上线后日常运维、备份、恢复演练和故障排查的最小操作标准。

## 1. 部署目标

生产环境建议采用：

- Django 单体应用。
- PostgreSQL 数据库。
- Nginx 反向代理和静态文件服务。
- 附件目录放在 Web 静态目录之外。
- 管理命令配合 Windows 任务计划程序或 Linux cron 执行后台任务。

MVP 阶段不要求 Celery、Redis、对象存储或独立 API 服务。只有当后台任务耗时、附件规模或移动端/API 需求明显增长时，再按主设计文档扩展。

### 2.1 Windows 内网无域名配置示例

如果服务器只在公司内网使用，没有域名，建议使用固定 IP 或内网主机名访问，例如：

- 服务器 IP：`192.168.1.10`
- 内网访问地址：`http://192.168.1.10:8000/`
- 应用目录：`D:\ERP\app`
- 数据目录：`D:\ERP\data`

`.env` 可按下面配置：

```ini
DJANGO_ENV=production
DJANGO_SECRET_KEY=替换为50位以上高强度随机密钥
DJANGO_DEBUG=false
DJANGO_ALLOWED_HOSTS=192.168.1.10
DJANGO_SESSION_COOKIE_SECURE=false
DJANGO_CSRF_COOKIE_SECURE=false
DJANGO_CSRF_TRUSTED_ORIGINS=
DJANGO_SECURE_SSL_REDIRECT=false
DJANGO_SECURE_HSTS_SECONDS=0
DJANGO_SECURE_HSTS_INCLUDE_SUBDOMAINS=false
DJANGO_SECURE_HSTS_PRELOAD=false
DJANGO_LOGIN_URL=/login/
DJANGO_LOGIN_REDIRECT_URL=/
DJANGO_SESSION_COOKIE_AGE=28800
DJANGO_LOG_LEVEL=INFO

DB_ENGINE=postgres
POSTGRES_DB=erp_db
POSTGRES_USER=erp_app
POSTGRES_PASSWORD=替换为数据库强密码
POSTGRES_HOST=127.0.0.1
POSTGRES_PORT=5432

DJANGO_STATIC_ROOT=D:\ERP\app\staticfiles
ERP_ATTACHMENT_DIR=D:\ERP\data\attachments
ERP_BACKUP_DIR=D:\ERP\backups
DJANGO_LOG_DIR=D:\ERP\logs
ERP_ATTACHMENT_SCAN_COMMAND=
ERP_ATTACHMENT_SCAN_TIMEOUT=30
ERP_ATTACHMENT_SCAN_RISK_ACCEPTED_BY=系统管理员
ERP_INTRANET_HTTP_RISK_ACCEPTED_BY=系统管理员
ERP_BOOTSTRAP_ADMIN_PASSWORD=替换为初始管理员强密码
```

注意：

- `DJANGO_ALLOWED_HOSTS` 必须写服务器固定 IP 或内网主机名，生产模式不能写 `localhost`、`127.0.0.1` 或 `*`。
- 如果将来增加域名和 HTTPS，应移除 `ERP_INTRANET_HTTP_RISK_ACCEPTED_BY`，并把 Secure Cookie、SSL Redirect、HSTS 和 `DJANGO_CSRF_TRUSTED_ORIGINS` 改回 HTTPS 配置。
- 内网 HTTP 传输未加密，建议只允许可信内网访问，并在防火墙中限制访问来源网段。

生产规则：

- `DJANGO_ENV=production` 时必须设置 `DJANGO_SECRET_KEY`，否则应用会拒绝启动。
- `DJANGO_DEBUG` 必须为 `false`。
- `DJANGO_ENV=production` 时 `DB_ENGINE` 必须为 `postgres`，否则应用会拒绝启动，防止正式环境误连 SQLite。
- `DJANGO_ENV=production` 时 `POSTGRES_PASSWORD` 必须填写，否则应用会拒绝启动。
- `DJANGO_ALLOWED_HOSTS` 必须写正式域名；生产护栏会拒绝 `*`、`localhost`、`127.0.0.1` 和 `testserver`。
- `DJANGO_CSRF_TRUSTED_ORIGINS` 必须写正式 HTTPS 访问源，例如 `https://erp.example.com`。
- 正式 HTTPS 部署时，`DJANGO_SESSION_COOKIE_SECURE`、`DJANGO_CSRF_COOKIE_SECURE` 和 `DJANGO_SECURE_SSL_REDIRECT` 必须为 `true`。
- `DJANGO_SECURE_HSTS_SECONDS` 建议生产 HTTPS 环境设为 `31536000`；只有确认整个域名长期只走 HTTPS 后，才将 `DJANGO_SECURE_HSTS_PRELOAD=true`。
- 如果是 Windows 内网 HTTP 部署且没有域名，必须显式把上述 HTTPS 安全开关设为 `false` 或 `0`，并填写 `ERP_INTRANET_HTTP_RISK_ACCEPTED_BY=风险接受人姓名或岗位`。生产预检会把该风险接受人写入报告；未填写时 `production_preflight --strict` 会阻断上线。
- `DJANGO_STATIC_ROOT`、`ERP_ATTACHMENT_DIR`、`ERP_BACKUP_DIR`、`DJANGO_LOG_DIR` 和 `ERP_RELEASE_GATE_REPORT_FILE` 支持绝对路径；如填写相对路径，系统会按应用 `BASE_DIR` 解析，避免服务工作目录不同导致文件落到错误位置。
- `ERP_ATTACHMENT_DIR` 不能放到 `staticfiles` 目录下。
- `ERP_BACKUP_DIR` 不能放到附件目录、静态文件目录或应用代码目录中，建议放到独立数据盘或可备份目录。
- `ERP_ATTACHMENT_SCAN_COMMAND` 为空时附件登记为“无需扫描”；如服务器已安装杀毒工具，建议配置带 `{file}` 占位符的扫描命令，扫描返回码非 0 时系统会拒绝附件上传。
- 如果正式上线暂不启用附件扫描，必须填写 `ERP_ATTACHMENT_SCAN_RISK_ACCEPTED_BY`，生产预检会把风险接受人写入报告；未配置扫描命令且未填写风险接受人时会产生预警。
- `ERP_RELEASE_GATE_REPORT_FILE` 指向最近一次发布门禁 Markdown 报告；`ERP_RELEASE_GATE_MAX_AGE_HOURS` 控制报告有效期，默认 24 小时。`production_preflight --strict` 和 `/health/` 会读取该报告，避免使用过期门禁结果上线。
- `ERP_PRELAUNCH_BACKUP_MAX_AGE_HOURS`、`ERP_PRELAUNCH_BACKUP_VERIFY_MAX_AGE_HOURS`、`ERP_PRELAUNCH_RESTORE_DRILL_MAX_AGE_HOURS` 和 `ERP_PRELAUNCH_EVENT_PROCESS_MAX_AGE_MINUTES` 控制预上线报告中运维证据的新鲜度。
- `ERP_PENDING_EVENT_MAX_RETRIES`、`ERP_PENDING_EVENT_RETRY_BASE_MINUTES` 和 `ERP_PENDING_EVENT_RUNNING_TIMEOUT_MINUTES` 控制事务后事件失败重试和处理中超时回收。
- `ERP_BACKGROUND_JOB_RUNNING_TIMEOUT_MINUTES` 控制后台任务运行中超时回收；超过该时间的同类型旧任务会在启动新任务前自动标记失败并通知管理员。

## 3. 数据库准备

PostgreSQL 示例：

```sql
CREATE DATABASE erp_db ENCODING 'UTF8' LC_COLLATE 'C' LC_CTYPE 'C' TEMPLATE template0;
CREATE USER erp_app WITH PASSWORD '替换为数据库密码';
GRANT CONNECT ON DATABASE erp_db TO erp_app;
\c erp_db
GRANT USAGE, CREATE ON SCHEMA public TO erp_app;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO erp_app;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT USAGE, SELECT, UPDATE ON SEQUENCES TO erp_app;
```

如果数据库由 `erp_app` 用户创建，可简化权限配置，但不建议使用 PostgreSQL 超级用户运行应用。

## 4. 应用部署

### 4.0 方案 A：Windows 轻量安装器

Windows 内网服务器建议优先采用轻量安装器方式。该方式不把 ERP 编译成单个 exe，而是用 PowerShell 脚本完成部署、初始化和服务注册，便于排查和后续升级。

轻量安装器目录：

```text
installer\
  README.md
  download-prerequisites.ps1
  preflight-prerequisites.ps1
  setup-postgres-db.ps1
  install-erp.ps1
  register-windows-service.ps1
  register-scheduled-tasks.ps1
  create-desktop-shortcut.ps1
  templates\
    intranet.env.template
  packages\
```

开发机或有外网的电脑先下载安装包：

```powershell
.\installer\download-prerequisites.ps1
```

安装包会保存到：

```text
installer\packages\
```

该目录用于现场安装，不提交 Git。建议下载完成后，将整个项目目录复制到服务器。

服务器上以管理员 PowerShell 执行：

```powershell
Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
.\installer\preflight-prerequisites.ps1
.\installer\setup-postgres-db.ps1
.\installer\install-erp.ps1 -ServerHost 192.168.1.10
.\installer\register-windows-service.ps1
.\installer\register-scheduled-tasks.ps1
.\installer\create-desktop-shortcut.ps1 -ServerHost 192.168.1.10
```

`register-scheduled-tasks.ps1` 会把备份、备份校验、事务后事件处理、备份清理、恢复演练注册为 Windows 计划任务。计划任务通过 `installer\run-scheduled-task.ps1` 隐藏后台运行，并把日志写入 `logs\scheduled-*.log`，不会在服务器桌面每 5 分钟弹出 CMD 黑色窗口。

如果已执行 `.\installer\build-launchers.ps1`，项目根目录会生成两个双击入口：

```text
ERP-Setup.exe
ERP-Uninstall.exe
```

说明：

- `ERP-Setup.exe` 是安装启动器，会以管理员权限调用 `installer\erp-setup-launcher.ps1`。
- `ERP-Uninstall.exe` 是卸载启动器，会删除 ERP Windows 服务和计划任务，但不删除数据库、附件、备份和日志。
- 这两个 exe 只是小型启动器，不包含全部 ERP 代码和安装包。部署服务器时必须复制完整项目目录，尤其是 `installer\packages\`。
- 如需重新生成这两个入口，在项目根目录执行 `.\installer\build-launchers.ps1`。

其中 `192.168.1.10` 替换为服务器固定 IP 或内网主机名。安装完成后访问：

```text
http://192.168.1.10:8000/
```

如果服务器尚未安装 Python、PostgreSQL、Git 或 NSSM，可从 `installer\packages\` 中手动安装对应安装包，再重新执行 `preflight-prerequisites.ps1`。

轻量安装器默认部署目录：

```text
D:\ERP\app
D:\ERP\data\attachments
D:\ERP\backups
D:\ERP\logs
```

如需改路径，可给 `install-erp.ps1` 和 `register-windows-service.ps1` 传入 `-InstallDir` 等参数。

```powershell

.\.venv\Scripts\python -m pip install --upgrade pip
.\.venv\Scripts\pip install -r requirements.txt

.\.venv\Scripts\python manage.py release_gate --operator admin --include-deploy-check --include-tests --include-production-preflight --report-file docs\latest-release-gate-report.md
.\.venv\Scripts\python manage.py business_smoke_test --operator admin
.\.venv\Scripts\python manage.py backup_daily
.\.venv\Scripts\python manage.py verify_backups
.\.venv\Scripts\python manage.py restore_drill
.\.venv\Scripts\python manage.py process_pending_events
.\.venv\Scripts\python manage.py record_release 2026.06.11.1 --summary "首次预上线发布" --released-by admin
.\.venv\Scripts\python manage.py prelaunch_report --strict --bootstrap-username admin --release-version 2026.06.11.1 --report-file docs\prelaunch-acceptance-report.md --deployment-runbook-file docs\deployment-runbook-2026.06.11.1.md
```

Linux 示例：

```bash

python manage.py deployment_runbook --host erp.example.com --operator admin --release-version 2026.06.11.1 --summary "首次预上线发布" --output-file docs/deployment-runbook-2026.06.11.1.md

python manage.py check --deploy
python manage.py release_gate --operator admin --include-deploy-check --include-tests --include-production-preflight --report-file docs/latest-release-gate-report.md
python manage.py business_smoke_test --operator admin
python manage.py backup_daily
python manage.py verify_backups
python manage.py restore_drill
python manage.py process_pending_events
python manage.py record_release 2026.06.11.1 --summary "首次预上线发布" --released-by admin
python manage.py prelaunch_report --strict --bootstrap-username admin --release-version 2026.06.11.1 --report-file docs/prelaunch-acceptance-report.md --deployment-runbook-file docs/deployment-runbook-2026.06.11.1.md
```

说明：

- 当前项目使用 Django 6.0.6，建议生产 Python 版本与开发环境保持一致。
- 首次上线前必须运行 `migrate`，并确认 `makemigrations --check --dry-run` 无新迁移。
- `check --deploy` 可能提示 HTTPS、安全 Header 等部署建议，应结合反向代理配置处理。
- `deployment_runbook --output-file ...` 应在正式部署前先执行并归档，生成的清单会包含后续预上线验收命令。
- `prelaunch_report --strict` 应放在 `backup_daily`、`verify_backups`、`restore_drill`、`process_pending_events` 和 `record_release` 之后执行，因为它会把成功备份、成功备份校验、成功恢复演练、事务后事件处理和发布记录作为上线签字证据。建议同时传入 `--release-version <版本号>` 和 `--deployment-runbook-file <清单路径>`，强制校验本次发布记录和部署命令清单，而不是读取任意历史证据。

### 4.1 首次管理员初始化

首次部署完成迁移后，执行幂等初始化命令创建或修复初始超级管理员和权限管理员角色。

Windows PowerShell 示例：

```powershell
$env:ERP_BOOTSTRAP_ADMIN_PASSWORD="替换为强密码"
.\.venv\Scripts\python manage.py bootstrap_admin --username admin --email admin@example.com --display-name 系统管理员 --password-env ERP_BOOTSTRAP_ADMIN_PASSWORD --noinput
.\.venv\Scripts\python manage.py bootstrap_admin --username admin --check-only
Remove-Item Env:\ERP_BOOTSTRAP_ADMIN_PASSWORD
.\.venv\Scripts\python manage.py production_preflight --strict
```

Linux 示例：

```bash
ERP_BOOTSTRAP_ADMIN_PASSWORD='替换为强密码' \
  .venv/bin/python manage.py bootstrap_admin \
  --username admin \
  --email admin@example.com \
  --display-name 系统管理员 \
  --password-env ERP_BOOTSTRAP_ADMIN_PASSWORD \
  --noinput
.venv/bin/python manage.py bootstrap_admin --username admin --check-only
.venv/bin/python manage.py production_preflight --strict
```

规则：

- 命令会自动补齐默认权限、`permission-admin` 权限管理员角色和初始超级管理员账号。
- 命令可以重复执行；用户已存在时默认只修复账号状态、超级管理员标记和角色关系，不覆盖原密码。
- `bootstrap_admin --check-only` 只验证初始化结果，不创建或修改数据，适合放在部署验收脚本中。
- 只有显式增加 `--reset-password` 时才会重置已存在用户的密码。
- 生产环境管理员密码长度至少 12 位，且不能包含用户名。
- `production_preflight --strict` 会自动检查生产安全开关、数据库连接、迁移状态、附件/备份/日志目录可写、附件/备份/日志/静态目录隔离、HTTPS 安全配置、静态文件、初始超级管理员、权限管理员角色、附件扫描配置、失败或卡住后台任务、失败或卡住事务后事件和最近发布门禁报告；有失败或预警时返回非 0。
- `release_gate --include-production-preflight --report-file ...` 内部执行生产预检时会跳过旧门禁报告检查，避免检查当前尚未生成的报告；整轮门禁成功后会立即写出新的门禁报告，后续 `/health/` 和单独执行 `production_preflight --strict` 会读取这份报告。

## 5. 启动应用

Windows 推荐用 Waitress 托管 WSGI，当前 `requirements.txt` 已锁定 `waitress` 版本。预上线可先用命令行启动，正式长期运行建议再交给 NSSM 或 Windows 服务管理器。

Windows PowerShell 示例：

```powershell
cd D:\ERP\app
.\.venv\Scripts\waitress-serve.exe --listen=127.0.0.1:8000 --threads=8 config.wsgi:application
```

Windows 内网无域名、直接用服务器 IP 访问时，可让 Waitress 监听内网网卡：

```powershell
cd D:\ERP\app
.\.venv\Scripts\waitress-serve.exe --listen=0.0.0.0:8000 --threads=8 config.wsgi:application
```

然后在 Windows 防火墙中只允许内网网段访问 TCP 8000。示例：

```powershell
New-NetFirewallRule -DisplayName "ERP Intranet 8000" -Direction Inbound -Protocol TCP -LocalPort 8000 -Action Allow -RemoteAddress 192.168.0.0/16
```

如果需要长期后台运行，建议用 NSSM 或 Windows 服务管理器把 Waitress 注册成服务；不要用 `runserver` 做正式生产服务。

Linux 推荐 Gunicorn，但当前 `requirements.txt` 未包含 Gunicorn；正式选择后应加入依赖锁定。示例：

```bash
pip install gunicorn
gunicorn config.wsgi:application --bind 127.0.0.1:8000 --workers 3 --timeout 120
```

如果暂时只在内网试运行，可使用：

```powershell
.\.venv\Scripts\python manage.py runserver 127.0.0.1:8000
```

注意：`runserver` 仅用于内网验收或临时演示，不作为正式生产服务。

## 6. Nginx 反向代理建议

最小配置示例：

```nginx
server {
    listen 80;
    server_name erp.example.com;

    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl http2;
    server_name erp.example.com;

    ssl_certificate /etc/nginx/ssl/erp.example.com.crt;
    ssl_certificate_key /etc/nginx/ssl/erp.example.com.key;

    client_max_body_size 50m;

    location /static/ {
        alias /srv/erp/app/staticfiles/;
    }

    location /media/ {
        return 404;
    }

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto https;
    }
}
```

附件下载必须继续走 Django 权限校验，不允许直接暴露附件目录。后续如附件变大，可升级为 Nginx `X-Accel-Redirect`，由 Django 校验权限后交给 Nginx 发送文件。

## 8. 附件安全扫描

附件上传默认会检查文件扩展名、大小、存储路径和来源单据权限。若生产环境具备杀毒引擎，可通过环境变量启用外部扫描命令：

```ini
ERP_ATTACHMENT_SCAN_COMMAND=clamscan --no-summary {file}
ERP_ATTACHMENT_SCAN_TIMEOUT=30
```

Windows 如使用 Microsoft Defender，可按实际服务器路径配置 `MpCmdRun.exe`，示例：

```ini
ERP_ATTACHMENT_SCAN_COMMAND="C:\Program Files\Windows Defender\MpCmdRun.exe" -Scan -ScanType 3 -File {file}
ERP_ATTACHMENT_SCAN_TIMEOUT=60
```

规则：

- `{file}` 会被替换为附件在服务器上的绝对路径；如果命令未包含 `{file}`，系统会把文件路径追加到命令末尾。
- 扫描命令返回码为 0 时附件登记为“通过”。
- 扫描命令返回码非 0、超时或无法执行时，附件上传会失败，系统不会登记附件记录。
- 如果公司暂未部署杀毒引擎，可保持 `ERP_ATTACHMENT_SCAN_COMMAND` 为空，但必须在 `.env` 中填写 `ERP_ATTACHMENT_SCAN_RISK_ACCEPTED_BY=风险接受人姓名或岗位`，并在上线验收清单中签字确认。

## 10. 回滚和恢复原则

应用发布失败：

1. 停止新版本应用进程。
2. 切回上一版代码目录或部署包。
3. 使用上一版依赖环境启动服务。
4. 如果本次发布已执行不可逆迁移，先不要直接回滚数据库，必须根据迁移内容单独评估。

数据异常：

1. 立即停止相关业务操作入口，避免继续写入。
2. 在 `/health/`、后台任务和日志中确认异常范围。
3. 使用 `verify_backups` 找到最近可用备份。
4. 先执行 `restore_drill` 验证备份可读。
5. 在独立恢复环境导入 `database.json` 和附件目录，核对数据后再决定是否恢复生产。
