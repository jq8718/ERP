# 核心服务方法规格

本文档定义 ERP 第一阶段核心服务方法的输入、输出、事务边界、锁定对象、错误码和后置事件。

原则：

- 页面视图只负责收集输入、调用服务方法、展示结果。
- 库存、核销、审批、单号、红冲、余额处理必须通过服务层统一入口。
- 所有核心服务方法必须进行权限校验、状态校验和后端重新计算。
- 涉及库存、收付款核销、审批状态、单号生成的方法必须使用数据库事务。
- 可延后动作写入 `pending_events`，由后台任务处理。

## 1. 通用服务返回结构

建议服务方法统一返回 `ServiceResult`。

```python
@dataclass
class ServiceResult:
    success: bool
    error_code: str | None = None
    message: str = ""
    data: dict = field(default_factory=dict)
    next_action: str | None = None
```

成功示例：

```python
ServiceResult(
    success=True,
    message="销售订单审核通过",
    data={"sales_order_id": 123, "sales_order_no": "SO202606080001"},
    next_action="view_detail",
)
```

失败示例：

```python
ServiceResult(
    success=False,
    error_code="STOCK_NOT_ENOUGH",
    message="库存不足，无法出库",
    data={"material_code": "FG001", "required_qty": "100", "available_qty": "60"},
)
```

## 2. 通用错误码

| 错误码 | 使用场景 |
| --- | --- |
| `AUTH_NO_PERMISSION` | 无权限 |
| `AUTH_SECOND_VERIFY_REQUIRED` | 需要二次验证 |
| `STATE_INVALID_TRANSITION` | 当前状态不能执行动作 |
| `STATE_ALREADY_PROCESSED` | 重复处理或幂等命中 |
| `DOC_NOT_FOUND` | 单据不存在 |
| `DOC_VERSION_CONFLICT` | 乐观锁冲突 |
| `STOCK_NOT_ENOUGH` | 库存不足 |
| `STOCK_BATCH_LOCKED` | 批次冻结 |
| `STOCK_NEGATIVE_NOT_ALLOWED` | 库存不能为负 |
| `BOM_NOT_ENABLED` | BOM 未启用 |
| `BOM_UNIT_CONVERSION_MISSING` | 单位换算缺失 |
| `APPROVAL_RULE_MISSING` | 审批规则缺失 |
| `APPROVAL_COMMENT_REQUIRED` | 驳回原因必填 |
| `PAYMENT_ALLOCATION_OVER` | 核销超额 |
| `PAYMENT_REVERSAL_OVER` | 红冲超额 |
| `PAYMENT_CREDIT_BALANCE_NOT_ENOUGH` | 待处理余额不足 |
| `SAMPLE_RETURN_QTY_OVER` | 借样归还数量超额 |
| `FILE_TYPE_NOT_ALLOWED` | 附件类型不允许 |
| `SYSTEM_CONFIG_MISSING` | 系统配置缺失 |

## 3. 通用事务规则

### 3.1 锁定顺序

库存相关锁定顺序：

```text
material_id ASC
location_id ASC
batch_id ASC
```

核销相关锁定顺序：

```text
customer/supplier
target orders/receipts by id ASC
payment/receipt document
allocations
```

### 3.2 幂等参数

所有关键 `POST` 动作建议传入 `idempotency_key`：

```python
idempotency_key: str
```

适用动作：

- 确认出库
- 确认入库
- 确认收款 / 付款
- 红冲
- 生成采购需求
- 借样归还
- 处理待处理余额

### 3.3 事务后事件

事务内写入 `pending_events`，常见事件：

| 事件 | 触发 |
| --- | --- |
| `purchase_received` | 进货确认入库 |
| `sales_shipped` | 销售出库确认 |
| `production_received` | 生产入库确认 |
| `sample_returned` | 借样归还确认 |
| `payment_confirmed` | 收款 / 付款确认 |
| `payment_reversed` | 收款 / 付款红冲 |
| `stock_count_adjusted` | 盘点调整 |

## 4. 销售与库存服务

### 4.1 `confirm_sales_order`

用途：审批通过销售订单，锁定 BOM，检查库存，生成欠料提醒。

输入：

```python
confirm_sales_order(
    sales_order_id: int,
    operator_id: int,
    approval_id: int | None = None,
    comment: str = "",
)
```

前置校验：

- 操作人有销售订单审核权限。
- 销售订单状态为 `pending_approval`。
- 订单头客户、收件信息、明细行、数量、价格已完整。
- 每个成品明细关联成品编码。
- 成品必须有启用 BOM；没有 BOM 的明细进入 `pending_bom`。

事务：

- 锁定销售订单和明细。
- 锁定需要读取的 BOM 版本。
- 锁定或创建欠料提醒记录。

处理步骤：

1. 校验订单状态和权限。
2. 对每个明细锁定当前启用 BOM。
3. 执行库存齐套检查。
4. 成品库存充足的明细标记 `sufficient`。
5. 需要生产且必需子件齐套的明细标记 `kitted`。
6. 必需子件缺料的明细生成 `shortage_alerts`。
7. 汇总订单头状态。
8. 写审批日志和审计日志。

输出：

```python
{
    "sales_order_id": 123,
    "status": "confirmed",
    "line_results": [
        {"item_id": 1, "inventory_check_status": "sufficient"},
        {"item_id": 2, "inventory_check_status": "shortage"},
    ],
}
```

错误码：

- `AUTH_NO_PERMISSION`
- `STATE_INVALID_TRANSITION`
- `BOM_NOT_ENABLED`
- `BOM_UNIT_CONVERSION_MISSING`

后置事件：

- `sales_order_confirmed`
- `shortage_created`

### 4.2 `recheck_sales_order_inventory`

用途：BOM 补齐、采购入库、库存调整后重新检查销售订单明细齐套状态。

输入：

```python
recheck_sales_order_inventory(
    sales_order_item_ids: list[int],
    trigger: str,
    operator_id: int | None = None,
)
```

前置校验：

- 销售订单明细未作废。
- 明细已锁定 BOM 或可重新锁定 BOM。

事务：

- 锁定销售订单明细。
- 锁定对应欠料提醒。

处理步骤：

1. 重新计算 BOM 需求。
2. 比较可用库存。
3. 更新 `inventory_check_status`。
4. 更新欠料提醒状态：部分到货 / 已齐套 / 已关闭。
5. 齐套后生成可生产系统消息。

错误码：

- `BOM_NOT_ENABLED`
- `BOM_UNIT_CONVERSION_MISSING`

后置事件：

- `shortage_kitted`

### 4.3 `confirm_sales_shipment`

用途：确认销售出库，扣减成品库存，更新销售明细发货数量。

输入：

```python
confirm_sales_shipment(
    shipment_id: int,
    operator_id: int,
    idempotency_key: str,
)
```

前置校验：

- 操作人有销售出库权限。
- 出库单状态为 `pending_confirm`。
- 出库数量不超过订单未发货数量。
- 批次未冻结且库存足够。

事务：

- 锁定出库单和明细。
- 按固定顺序锁定 `inventory_batches` 和 `inventory`。
- 锁定销售订单明细。

处理步骤：

1. 校验幂等键。
2. 重新校验库存。
3. 扣减批次库存。
4. 更新 `inventory` 汇总。
5. 写 `inventory_transactions`。
6. 更新 `sales_order_items.shipped_qty` 和状态。
7. 更新订单头状态。
8. 确认出库单。

错误码：

- `STATE_INVALID_TRANSITION`
- `STOCK_NOT_ENOUGH`
- `STOCK_BATCH_LOCKED`
- `STOCK_NEGATIVE_NOT_ALLOWED`

后置事件：

- `sales_shipped`

## 5. 采购服务

### 5.1 `create_purchase_request_from_shortages`

用途：从欠料提醒生成采购需求。

输入：

```python
create_purchase_request_from_shortages(
    shortage_alert_ids: list[int],
    operator_id: int,
    merge_mode: str,
    idempotency_key: str,
)
```

前置校验：

- 操作人有生成采购需求权限。
- 欠料状态为 `unprocessed` 或允许重新生成。

事务：

- 锁定欠料提醒。
- 创建采购需求和明细。

处理步骤：

1. 校验欠料提醒未重复生成。
2. 按物料、供应商、需求日期合并。
3. 创建 `purchase_requests` 采购需求单头。
4. 创建 `purchase_request_items` 采购需求明细，记录来源欠料提醒和来源销售订单明细。
5. 更新欠料状态为 `purchase_requested`。

错误码：

- `STATE_ALREADY_PROCESSED`
- `STATE_INVALID_TRANSITION`

### 5.2 `confirm_purchase_receipt`

用途：确认进货入库，生成批次，更新库存，写流水。

输入：

```python
confirm_purchase_receipt(
    purchase_receipt_id: int,
    operator_id: int,
    idempotency_key: str,
)
```

前置校验：

- 操作人有入库权限。
- 进货单状态为 `pending_receive` 或 `partial_received`。
- 合格数量、不良数量、单价已填写。

事务：

- 锁定进货单和明细。
- 按固定顺序锁定或新建库存记录。
- 锁定采购单明细。

处理步骤：

1. 校验幂等键。
2. 生成库存批次。
3. 更新 `inventory`。
4. 写库存流水。
5. 更新进货单入库状态。
6. 更新采购单明细 `received_qty`。
7. 写入 `pending_events.purchase_received`。

错误码：

- `STATE_INVALID_TRANSITION`
- `STOCK_NEGATIVE_NOT_ALLOWED`

后置事件：

- `purchase_received`，用于欠料重检和消息通知。

## 6. 库存服务

### 6.1 `confirm_stock_count_adjustment`

用途：确认盘点调整，根据盘点差异更新库存批次、当前库存和库存流水。

输入：

```python
confirm_stock_count_adjustment(
    stock_count_id: int,
    operator_id: int,
    idempotency_key: str,
)
```

前置校验：

- 操作人有库存调整或盘点审核权限。
- 盘点单状态为 `approved_pending_adjustment` 或允许调整状态。
- 盘点明细已录入实盘数量和差异原因。
- 如果盘点范围配置了冻结，冻结范围仍处于有效状态。
- 差异为 0 的明细不生成调整流水。

事务：

- 锁定盘点单和盘点明细。
- 按 `material_id`、`location_id`、`batch_id` 顺序锁定相关 `inventory_batches` 和 `inventory`。
- 锁定相关业务汇总记录。

处理步骤：

1. 校验幂等键。
2. 校验盘点单状态和权限。
3. 对每条差异明细计算 `difference_qty = counted_qty - book_qty`。
4. 盘盈时增加批次或创建盘点批次。
5. 盘亏时扣减指定批次或按盘点快照批次扣减。
6. 更新 `inventory` 汇总。
7. 写入 `inventory_transactions`，类型为 `stock_adjustment`。
8. 更新盘点单状态为 `adjusted`。
9. 写审计日志。
10. 写入 `pending_events.stock_count_adjusted`。

输出：

```python
{
    "stock_count_id": 100,
    "adjusted_items": 8,
    "total_gain_qty": "12.0000",
    "total_loss_qty": "3.0000",
}
```

错误码：

- `AUTH_NO_PERMISSION`
- `STATE_INVALID_TRANSITION`
- `STOCK_NEGATIVE_NOT_ALLOWED`
- `STATE_ALREADY_PROCESSED`

后置事件：

- `stock_count_adjusted`，用于业务汇总校验和管理员消息。

confirm_production_receipt(

前置校验：

- 入库数量不超过生产数量和允许超产范围。
- 生产入库单状态允许确认。

事务：

- 锁定生产入库单。
- 新增成品批次。
- 更新库存汇总。
- 更新生产指令 `received_qty` 和状态。

后置事件：

- `production_received`

## 8. 借样服务

### 8.1 `confirm_sample_return`

用途：确认借样归还入库，更新库存和借样状态。

输入：

```python
confirm_sample_return(
    sample_return_id: int,
    operator_id: int,
    idempotency_key: str,
)
```

前置校验：

- 操作人有借样归还或库存入库权限。
- 归还单状态为 `pending_confirm`。
- 归还数量不超过未归还且未转销售数量。
- 损坏、缺件、数量异常按审批规则处理。

事务：

- 锁定归还单和归还明细。
- 锁定借样单和 `sample_loan_items` 借样明细。
- 锁定或新增库存批次。
- 更新库存汇总。

处理步骤：

1. 校验幂等键。
2. 校验可归还数量。
3. 按样品状态决定入可用库存或待处理库存。
4. 写库存批次和库存流水 `sample_return_in`。
5. 更新 `sample_loan_items.returned_qty`。
6. 重算借样主状态和逾期标记。
7. 关闭已完成借样的逾期提醒。
8. 写入 `pending_events.sample_returned`。

输出：

```python
{
    "sample_return_id": 10,
    "sample_loan_id": 20,
    "sample_status": "part_returned",
    "overdue_status": "closed",
}
```

错误码：

- `SAMPLE_RETURN_QTY_OVER`
- `STATE_INVALID_TRANSITION`
- `STOCK_NEGATIVE_NOT_ALLOWED`

## 9. 财务服务

### 9.1 `confirm_customer_receipt`

用途：确认客户收款并写入核销明细。

输入：

```python
confirm_customer_receipt(
    receipt_id: int,
    allocations: list[dict],
    operator_id: int,
    idempotency_key: str,
)
```

`allocations` 示例：

```python
[
    {"sales_order_id": 1, "allocated_amount": "1000.00"},
    {"sales_order_id": 2, "allocated_amount": "500.00"},
]
```

前置校验：

- 操作人有收款确认权限。
- 收款单状态为 `pending_approval` 或 `approved_pending_confirm`。
- 核销总额不超过收款金额。
- 单个订单核销后不超过订单可核销金额。

事务：

- 锁定收款单。
- 锁定客户和目标销售订单。
- 锁定已有核销明细。

处理步骤：

1. 校验幂等键。
2. 后端重新计算订单可核销余额。
3. 写入核销明细。
4. 未分配金额生成或更新客户待处理余额。
5. 确认收款状态。
6. 写审计日志。

错误码：

- `PAYMENT_ALLOCATION_OVER`
- `STATE_INVALID_TRANSITION`

后置事件：

- `payment_confirmed`

### 9.2 `reverse_customer_receipt`

用途：对已确认客户收款进行红冲。

输入：

```python
reverse_customer_receipt(
    receipt_id: int,
    reversal_amount: Decimal,
    reason: str,
    operator_id: int,
    idempotency_key: str,
)
```

前置校验：

- 操作人有收款红冲权限。
- 原收款单状态为 `confirmed` 或 `part_reversed`。
- 红冲金额不超过可红冲金额。
- 原收款单核销明细可追溯。

事务：

- 锁定原收款单。
- 锁定原核销目标订单。
- 锁定原核销明细。

处理步骤：

1. 创建 `customer_receipt_reversals`。
2. 生成反向核销明细。
3. 重算订单已收金额和状态。
4. 更新原收款单状态为 `part_reversed` 或 `reversed`。
5. 如产生差额，生成客户待处理余额。
6. 写入 `pending_events.payment_reversed`。

错误码：

- `PAYMENT_REVERSAL_OVER`
- `STATE_INVALID_TRANSITION`
- `APPROVAL_RULE_MISSING`

### 9.3 `apply_customer_credit_balance`

用途：处理客户预收、超收、退款或转核销余额。

输入：

```python
apply_customer_credit_balance(
    credit_balance_id: int,
    action_type: str,
    amount: Decimal,
    operator_id: int,
    target_sales_order_id: int | None = None,
    reason: str = "",
    attachment_ids: list[int] | None = None,
    idempotency_key: str = "",
)
```

`action_type`：

- `to_advance`
- `allocate_to_order`
- `refund`
- `close`

前置校验：

- 余额状态允许处理。
- 金额不超过剩余余额。
- 核销目标订单属于同一客户。

事务：

- 锁定余额记录。
- 锁定目标订单或退款记录。
- 写余额流水。

错误码：

- `PAYMENT_CREDIT_BALANCE_NOT_ENOUGH`
- `PAYMENT_ALLOCATION_OVER`
- `STATE_INVALID_TRANSITION`

### 9.4 `confirm_supplier_payment`

与 `confirm_customer_receipt` 对称，目标单据为进货单或供应商对账单。

关键差异：

- 锁定供应商和进货单。
- 校验进货可付金额 = 进货应付金额 - 已核销付款金额 - 供应商退货金额。
- 未分配金额生成供应商预付款或超付待处理余额。

### 9.5 `reverse_supplier_payment`

与 `reverse_customer_receipt` 对称，来源单据为供应商付款单，目标为进货单。

### 9.6 `apply_supplier_credit_balance`

与 `apply_customer_credit_balance` 对称，目标单据为同供应商其他进货单。

## 10. 审批服务

### 10.1 `apply_approval_action`

用途：审批通过、驳回、转交、加签、退回修改。

输入：

```python
apply_approval_action(
    approval_id: int,
    action: str,
    operator_id: int,
    comment: str = "",
    target_user_id: int | None = None,
    second_verify_token: str | None = None,
)
```

`action`：

- `approve`
- `reject`
- `transfer`
- `add_approver`
- `return_to_edit`
- `withdraw`

前置校验：

- 操作人是当前审批人或有管理员审批权限。
- 驳回、退回修改必须填写原因。
- 高敏审批需要二次验证。

事务：

- 锁定审批单。
- 写审批日志。
- 如果审批链结束，通过来源业务服务执行业务状态变更。

处理步骤：

1. 校验审批人。
2. 校验动作合法。
3. 写 `approval_logs`。
4. 更新当前审批节点。
5. 如果最终通过，调用来源业务服务。
6. 生成系统消息。

错误码：

- `AUTH_NO_PERMISSION`
- `AUTH_SECOND_VERIFY_REQUIRED`

## 11. 系统服务

### 11.1 `void_document`

用途：统一作废未产生正式影响或允许作废的单据。

输入：

```python
void_document(
    doc_type: str,
    doc_id: int,
    reason: str,
    operator_id: int,
    idempotency_key: str,
)
```

前置校验：

- 单据存在。
- 单据状态允许作废。
- 已产生库存、财务或审批影响的单据不能直接作废，需要红冲、调整或反向单据。

错误码：

- `STATE_INVALID_TRANSITION`
- `AUTH_NO_PERMISSION`

### 11.2 `process_pending_events`

用途：处理事务提交后的持久化事件。

输入：

```python
process_pending_events(
    event_type: str | None = None,
    limit: int = 100,
)
```

处理步骤：

1. 查询 `pending_events` 中 `pending` 或到期重试事件。
2. 使用行级锁领取事件，并将事件临时标记为 `running`。
3. 根据 `event_type` 分发处理器。
4. 成功后标记 `success`，清空 `last_error` 和 `next_retry_at`。
5. 失败时记录错误、增加 `retry_count`，按退避时间设置 `next_retry_at`。

错误处理：

- 单个事件失败不影响其他事件。
- `ERP_PENDING_EVENT_MAX_RETRIES` 控制最大处理次数，默认 3 次。
- `ERP_PENDING_EVENT_RETRY_BASE_MINUTES` 控制退避基准分钟数，默认 5 分钟；第 1 次失败后 5 分钟重试，第 2 次失败后 10 分钟重试。
- 未到 `next_retry_at` 的失败事件不会被领取。
- `ERP_PENDING_EVENT_RUNNING_TIMEOUT_MINUTES` 控制 `running` 事件超时回收窗口，默认 30 分钟；超过该时间仍为 `running` 的事件会在下一次后台任务中重新领取。
- 超过最大重试次数后保持 `failed` 且不再设置 `next_retry_at`，后台任务失败告警通知管理员。
